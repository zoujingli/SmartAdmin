# 迁移与升级

## 迁移策略 🧱

- 开发期建表与基线维护使用 `migrations`
- 发布升级使用 release 快照机制，不依赖线上执行完整迁移历史

当前项目重点是“发布快照驱动升级”，不是把历史迁移脚本逐条在生产重放。

## 升级流程（推荐）🚀

1. 执行 `composer release:backup` 生成结构/数据快照
2. 执行 `composer release:upgrade:dry-run` 预览 SQL 与影响范围
3. 在已发布环境执行 `<binary> --self xadmin:release:upgrade --dry-run --json`
4. 重点检查 `ignore_tables` 与 `backup_tables` 命中情况
5. 确认风险后执行 `<binary> --self xadmin:release:upgrade`
6. 如需恢复受管数据，使用 `<binary> --self xadmin:release:restore --backup=<id>`
7. 如需补充执行迁移升级，先执行 `<binary> --self migrate:status` 和 `<binary> --self migrate --pretend --force`，确认后执行 `<binary> --self migrate --force`
8. 发布后做功能回归（登录、权限、菜单、核心 CRUD）

## 关键参数与语义 🔧

- `ignore_tables`：发布系统忽略的表
- `backup_tables`：发布前备份并按快照接管的表

这两组配置决定了“哪些表参与结构对比、哪些表会被发布快照接管”，是升级安全的核心开关。

## 与权限/菜单同步的关系 🔐

菜单和权限节点同步属于源码/CI 构建职责，通常在 `composer build` 前完成：

```bash
sh bin/start-swoole xadmin:menu:sync --details
sh bin/start-swoole xadmin:node:sync --details
```

因为当前权限模型依赖“注解 + 菜单 code + system_node 注册表”三者一致。已发布 Phar/SFX 二进制会隐藏 `xadmin:menu:sync` 和 `xadmin:node:sync`，正常升级不再提示生产用户执行；仅在权限注册表异常等极端修复时，才使用 `<binary> --self xadmin:menu:sync --dry-run --json` 或 `<binary> --self xadmin:node:sync --dry-run --json` 精确执行。

## 风险控制清单 ✅

- 生产升级前必须进行 dry-run
- 高风险结构变更需准备回滚与数据恢复方案
- 已发布二进制只保留前向迁移 `migrate`、`migrate:install`、`migrate:status`；不要给 `migrate` 附带 `--seed`；不暴露 `migrate:rollback`、`migrate:fresh`、`migrate:reset`、`migrate:refresh`、`db:seed`、`gen:*`、`xadmin:docs:check` 和 `xadmin:build:*`
- 升级前后对比关键统计：用户数、角色数、节点数、日志采集状态
- 发生异常时优先恢复快照，再定位差异 SQL
