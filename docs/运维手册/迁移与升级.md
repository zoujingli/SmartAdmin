# 迁移与升级

## 迁移策略 🧱

- 开发期建表与基线维护使用各应用插件的 `stc/migrations`，并由 `plugin.migration_root` 显式注册。
- 发布包安装和升级使用 release 安装包与运行备份，不依赖线上执行完整迁移历史。

当前项目重点是“开发期 migrate、正式包 restore”，不是把历史迁移脚本逐条在生产重放。

## 升级流程（推荐）🚀

1. 源码/CI 执行 `composer release:backup` 生成 `storage/extra/release` 安装包。
2. 执行 `composer release:restore:dry-run` 预览安装包恢复 SQL 与必要数据表。
3. 在已发布环境先执行 `<binary> --self xadmin:release:backup --with-data` 生成运行全量备份。
4. 执行 `<binary> --self xadmin:release:restore --install --dry-run --json` 评审 SQL。
5. 确认风险后执行 `<binary> --self xadmin:release:restore --install`；如确需破坏性同步再追加 `--force`。
6. 如需回滚数据库，使用 `<binary> --self xadmin:release:restore --with-data` 恢复最新全量运行备份。
7. 发布后做功能回归（登录、权限、菜单、核心 CRUD）。

## 关键参数与语义 🔧

- `backup_tables`：必要数据表；不加 `--with-data` 时只备份和恢复这些表的数据。
- `ignore_tables`：不进入必要数据的表；优先级高于 `backup_tables`。
- `xadmin:release:backup --install`：生成 Phar 内安装包，禁止与 `--with-data` 同用。
- `xadmin:release:backup --with-data`：生成运行全量备份，默认路径 `runtime/backup/<timestamp>/`。
- `xadmin:release:restore --install`：从 Phar 内 `storage/extra/release` 恢复结构和必要数据。
- `xadmin:release:restore --with-data`：从最新运行全量备份恢复结构和全部数据。

结构快照包含当前数据库全部表结构；“必要数据”由 `backup_tables` 决定。

## 与权限/菜单同步的关系 🔐

菜单和权限节点同步属于源码/CI 构建职责，通常在 `composer build` 前完成：

```bash
./bin/smart xadmin:menu:sync --details
./bin/smart xadmin:node:sync --details
```

因为当前权限模型依赖“注解 + 菜单 code + system_node 注册表”三者一致。已发布 Phar/SFX 二进制会隐藏 `xadmin:menu:sync` 和 `xadmin:node:sync`，正常升级不再提示生产用户执行；仅在权限注册表异常等极端修复时，才使用 `<binary> --self xadmin:menu:sync --dry-run --json` 或 `<binary> --self xadmin:node:sync --dry-run --json` 精确执行。

## 风险控制清单 ✅

- 生产升级前必须进行 dry-run。
- 高风险结构变更需先执行 `xadmin:release:backup --with-data` 并确认可恢复。
- 已发布二进制不注册 `migrate`、`migrate:install`、`migrate:status`、`migrate:rollback`、`migrate:fresh`、`migrate:reset`、`migrate:refresh`、`db:seed`、`gen:*`、`xadmin:docs:check`、`xadmin:build:*`、`xadmin:plugin:*` 和历史发布升级专用命令。
- 升级前后对比关键统计：用户数、角色数、节点数、日志采集状态。
- 发生异常时优先使用最新全量运行备份恢复，再定位差异 SQL。
