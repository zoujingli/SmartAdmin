# 部署与发布

## 本地与生产差异 🧭

- 本地以源码 + 开发依赖运行，允许使用 `migrate`、`xadmin:plugin:*`、docs/build/dev 命令。
- 生产推荐使用构建产物（二进制/SFX）运行，只保留安装、备份、恢复、前端发布、密码重置和必要业务运行入口；构建、docs、插件管理等源码命令不应出现。

## 一键构建 📦

```bash
composer web:build
composer build
```

`composer build` 会复用并校验已有 `web/dist`，再串联菜单/节点同步、release 安装包生成、生产依赖安装、容器缓存预编译与打包流程；打包前需先执行 `composer web:build` 或由 CI 提供等价前端产物。

构建编排统一由 `.php-sfx-packer.php build` 承载；该脚本同时提供 `precompile`、`audit`、`pack` 模式，便于 CI 或人工分段验证。构建产物会生成 `build/system`、`build/system-linux-x64`、`build/system-linux-a64`、`build/system-macos-a64`，其中默认 `build/system` 与 Linux x64 产物一致；审计通过后还会同步 `build/upgrade/system`、`build/upgrade/system-linux-x64` 和 `build/upgrade/system-linux-a64`，供一键升级模板使用。

`build/upgrade/start.sh` 是 Docker/1Panel 场景的一键替换升级模板，默认上传同目录的 `system` 到远端 `/tmp/system`，替换宿主机程序文件后在容器内执行 `xadmin:release:restore --install` 与 `xadmin:website:publish`，最后重启容器。脚本不执行全量备份、dry-run 或 `--force`；服务器、远端路径、容器名和密码都应通过环境变量或交互输入提供，禁止把真实账号密码写入仓库。

## Swoole 基库 🧱

仓库内置精简 PHP 8.4 + Swoole 6.2 SFX 基库，覆盖 Hyperf、MySQL、SQLite、Redis、Zip、XML、GD 和 OPcache 等运行能力；打包审计会校验必要扩展存在并确认未加载项目未使用的重型扩展。

如需按业务自定义扩展或重新构建基库，可参考 [zoujingli/phpsfx](https://github.com/zoujingli/phpsfx)。本仓库不内置 `tools/phpsfx/`，只交付已验证的 `bin/swoole-*` 运行时。

## 二进制命令约定 ▶️

Swoole CLI 的 SFX 包装层需要先切到追加的应用 Phar，二进制模式下统一使用：

```bash
./build/system-linux-x64 --self start
./build/system-linux-x64 --self xadmin:release:install --dry-run --json
./build/system-linux-x64 --self xadmin:release:install
./build/system-linux-x64 --self xadmin:release:backup --with-data
./build/system-linux-x64 --self xadmin:release:restore --with-data --dry-run --json
./build/system-linux-x64 --self xadmin:release:restore --install --dry-run --json
./build/system-linux-x64 --self xadmin:password:reset --default --json
```

不要直接执行 `./build/system-linux-x64 start`，否则命令会停留在 SFX 运行时层，无法进入应用命令解析。

Phar 内前端资源以 `storage/extra/web-dist.zip` 保存，首次 `start` 时若 `public/index.html` 不存在会自动发布；手动发布仅作为维护动作使用：

```bash
./build/system-linux-x64 --self xadmin:website:publish --dry-run
./build/system-linux-x64 --self xadmin:website:publish
```

Phar 内数据库安装包位于 `storage/extra/release/*`，`xadmin:release:install` 会执行 `restore --install` + 前端 publish。生产二进制的 `list` 不应显示 `migrate`、rollback/fresh/reset/refresh、seed、生成器、docs、build、`xadmin:plugin:*` 和历史发布升级专用命令；确需生产执行的插件业务命令必须在插件文档中说明用途和风险。

## 源码/CI 发布命令 🛠️

```bash
composer release:check
composer release:backup
composer release:restore:dry-run
composer release:snapshot
composer docs:check
./bin/smart.php xadmin:menu:sync --dry-run --json
./bin/smart.php xadmin:node:sync --dry-run --json
```

这些命令只在源码或 CI 中运行。`composer release:backup` 等价于 `xadmin:release:backup --install`，写入 `storage/extra/release` 供 Phar 打包；默认运行备份命令 `xadmin:release:backup` 则写入 `runtime/backup/<timestamp>/`。SmartAdminLibrary 提供的 `xadmin:plugin:*` 插件打包、安装、移除、备份恢复，以及菜单/节点清单同步，也都属于源码/CI 构建职责。已发布环境首次安装和升级使用 `xadmin:release:install` 或 `xadmin:release:restore --install`，不在 Phar 内执行 migrate、插件安装更新、rollback/fresh 或构建命令。

## SQLite 二进制模拟部署记录 🧪

2026-05-26 使用 SQLite 从空目录模拟正式二进制部署，目标是验证安装不依赖 `migrate`，发布包直接从 Phar 内 `storage/extra/release` 恢复：

```bash
# 开发库准备：本地开发版本仍用 migrate 建库
SRC=/tmp/smartadmin-sqlite-build-source-20260526-release
DB_DRIVER=sqlite DB_DATABASE=$SRC/runtime/system.db CACHE_DRIVER=file \
APP_ENV=prod APP_DEBUG_XSQL=false APP_DEBUG_XLOG=false \
./bin/swoole-macos-a64 bin/hyperf.php migrate --force --no-ansi

# 基于 SQLite 开发库打包，composer build 会生成 storage/extra/release 安装包并打入 Phar
DB_DRIVER=sqlite DB_DATABASE=$SRC/runtime/system.db CACHE_DRIVER=file \
APP_ENV=prod APP_DEBUG_XSQL=false APP_DEBUG_XLOG=false \
COMPOSER_ALLOW_SUPERUSER=1 composer build
```

部署目录只需要二进制和 `.env`，不再复制外置数据库安装包目录：

```bash
DEPLOY=/tmp/smartadmin-sqlite-deploy-20260526-release
mkdir -p $DEPLOY
cp build/system-macos-a64 $DEPLOY/system-macos-a64
cp build/.env.example $DEPLOY/.env
```

`.env` 关键项：

```ini
APP_ENV=prod
APP_WORKER_PORT=19501
APP_DEBUG_XSQL=false
APP_DEBUG_XLOG=false
APP_TITLE="SmartAdmin SQLite 模拟"
APP_API_URL=/
DB_DRIVER=sqlite
DB_DATABASE=runtime/system.db
CACHE_DRIVER=file
```

正式初始化和运行备份恢复命令：

```bash
cd /tmp/smartadmin-sqlite-deploy-20260526-release
./system-macos-a64 --self list --raw
./system-macos-a64 --self xadmin:release:install --dry-run --json
./system-macos-a64 --self xadmin:release:install
./system-macos-a64 --self xadmin:release:backup --with-data --json
./system-macos-a64 --self xadmin:release:restore --with-data --dry-run --json
./system-macos-a64 --self xadmin:release:restore --with-data --json
./system-macos-a64 --self xadmin:password:reset --default --json
```

期望结果：

- 二进制命令列表没有 `migrate`、`xadmin:plugin:*` 和历史发布升级专用命令。
- `xadmin:release:install` 从 Phar 内安装包恢复结构与必要数据，并发布前端资源到 `public/`。
- `xadmin:release:backup --with-data` 在 `runtime/backup/<timestamp>/` 生成结构 + 全量数据备份，并更新 `runtime/backup/latest`。
- `xadmin:release:restore --with-data` 能从最新全量运行备份恢复；若最新备份不是 `with_data=true` 会拒绝。
- `xadmin:password:reset --default --json` 可恢复默认管理员密码。

SQLite 适合单机演示和本地模拟；生产多实例部署仍推荐 MySQL + Redis，并按同样的 release install/restore 流程初始化和升级。

## 发布前检查（建议门禁）✅

- `composer release:check` 通过。
- 菜单与权限节点同步结果无异常（必要时查看 `--details`）。
- 关键配置已在 `.env` 明确，不依赖本地默认值。
- 文档和更新日志已同步（接口/权限/迁移变更必须记录）。

## 部署建议 🚀

- 发布前先执行 `release:check`。
- 变更数据库结构前保留 `xadmin:release:backup --with-data` 运行全量备份并评估恢复方案。
- 使用 `.env` 管理运行参数，避免写死在代码中。

## 发布后验证 🔍

- 登录、刷新 token、退出链路是否正常。
- 菜单树与按钮权限是否与角色一致。
- `system/data` 看板统计是否正常刷新。
- 日志系统是否正常记录请求与关键写操作。
