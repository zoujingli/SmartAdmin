# 部署与发布

## 本地与生产差异 🧭

- 本地以源码 + 开发依赖运行
- 生产推荐使用构建产物（二进制/SFX）运行

## 一键构建 📦

```bash
composer web:build
composer build
```

`composer build` 会复用并校验已有 `web/dist`，再串联菜单/节点同步、生产依赖安装、发布快照、容器缓存预编译与打包流程；打包前需先执行 `composer web:build` 或由 CI 提供等价前端产物。

构建编排统一由 `.php-sfx-packer.php build` 承载；该脚本同时提供 `precompile`、`audit`、`pack` 模式，便于 CI 或人工分段验证。构建产物会生成 `build/system`、`build/system-linux-x64`、`build/system-linux-a64`、`build/system-macos-a64`，其中默认 `build/system` 与 Linux x64 产物一致。

## Swoole 基库 🧱

仓库内置精简 PHP 8.4 + Swoole 6.2 SFX 基库，覆盖 Hyperf、MySQL、Redis、Zip、XML、GD 和 OPcache 等运行能力；打包审计会校验必要扩展存在并确认未加载项目未使用的重型扩展。

如需按业务自定义扩展或重新构建基库，可参考 [zoujingli/phpsfx](https://github.com/zoujingli/phpsfx)。本仓库不内置 `tools/phpsfx/`，只交付已验证的 `bin/swoole-*` 运行时。

## 二进制命令约定 ▶️

Swoole CLI 的 SFX 包装层需要先切到追加的应用 Phar，二进制模式下统一使用：

```bash
./build/system-linux-x64 --self start
./build/system-linux-x64 --self xadmin:release:upgrade --dry-run --json
./build/system-linux-x64 --self xadmin:release:upgrade
./build/system-linux-x64 --self xadmin:release:restore --backup=<id>
./build/system-linux-x64 --self migrate:status
./build/system-linux-x64 --self migrate --pretend --force
./build/system-linux-x64 --self migrate --force
```

不要直接执行 `./build/system-linux-x64 start`，否则命令会停留在 SFX 运行时层，无法进入应用命令解析。

Phar 内前端资源以 `storage/extra/web-dist.zip` 保存，首次 `start` 时若 `public/index.html` 不存在会自动发布；手动发布仅作为维护动作使用：

```bash
./build/system-linux-x64 --self xadmin:site:publish --dry-run
./build/system-linux-x64 --self xadmin:site:publish
```

该命令会跳过动态 `_app.config.js`。生产二进制的 `list` 只显示运行、前向迁移和必要维护命令；Hyperf 3.2 前向迁移命令是 `migrate`（不是 `migrate:run`），发布包保留 `migrate`、`migrate:install`、`migrate:status` 用于补充升级，不显示 rollback/fresh/reset/refresh、seed、生成器、docs、build、plugin 管理和 release backup；`xadmin:menu:sync`、`xadmin:node:sync` 在 Phar 模式隐藏，仅保留极端修复时精确执行。

## 源码/CI 发布命令 🛠️

```bash
composer release:check
composer release:backup
composer release:upgrade:dry-run
composer release:snapshot
composer docs:check
sh bin/start-swoole xadmin:menu:sync --dry-run --json
sh bin/start-swoole xadmin:node:sync --dry-run --json
```

这些命令只在源码或 CI 中运行。SmartAdminLibrary 提供的 `xadmin:plugin:*` 插件打包、安装、移除、备份恢复，以及菜单/节点清单同步，也都属于源码/CI 构建职责；插件 backup 默认只备份代码，`--with-data` 才包含插件自有表，remove 前自动备份必须带数据。已发布环境优先使用 release 快照和二进制 `xadmin:release:upgrade`；如需迁移补充升级，仅使用 `migrate --pretend --force` 和 `migrate --force`，不要附带 `--seed`，也不要在 Phar 内执行插件安装更新、rollback/fresh 或构建命令。

## 发布前检查（建议门禁）✅

- `composer release:check` 通过
- 菜单与权限节点同步结果无异常（必要时查看 `--details`）
- 关键配置已在 `.env` 明确，不依赖本地默认值
- 文档和更新日志已同步（接口/权限/迁移变更必须记录）

## 部署建议 🚀

- 发布前先执行 `release:check`
- 变更数据库结构前保留快照并评估回滚方案
- 使用 `.env` 管理运行参数，避免写死在代码中

## 发布后验证 🔍

- 登录、刷新 token、退出链路是否正常
- 菜单树与按钮权限是否与角色一致
- `system/data` 看板统计是否正常刷新
- 日志系统是否正常记录请求与关键写操作
