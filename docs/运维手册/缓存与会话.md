# 缓存与会话

## 缓存驱动与定位 🧠

当前系统基于 Hyperf Cache 抽象，支持 `file`、`redis`、`memory` 等驱动切换。

建议：

- 开发环境可用 `file`
- 生产环境与多实例部署优先 `redis`
- 缓存驱动切换后需验证权限缓存与在线会话统计是否正常

## 权限缓存机制（AuthCacheService）🔐

权限缓存采用“**版本号失效**”策略，而不是全量遍历删除：

- 全局版本键：`system:perm:ver:global`
- 用户版本键：`system:perm:ver:user:{userId}`
- 用户权限键：`system:perm:user:{userId}:g{gv}:u{uv}`

设计收益：

- 任一版本变化都会让旧缓存键自然失效
- 避免大规模 delete 带来的峰值开销
- 便于在高并发场景做权限变更

典型触发场景：

- 用户角色变化：提升用户版本
- 角色节点变化：提升全局版本
- 手动清理缓存：删除当前键并提升版本，同时清理协程上下文缓存

## 在线会话机制（OnlineUserService）🟢

在线会话索引键：`system:online-users`

核心行为：

- 登录后按 token 生成会话索引（key 为 token 的 sha1）
- 会话条目包含：用户、场景、最后活跃时间、过期时间
- 自动清理过期会话并回写索引
- 支持单 token 下线、单用户全会话下线、全站清空

这套实现不依赖 Redis 集合类型，因此对 `file/redis/memory` 驱动都可工作。

## 运行与排查建议 🛠️

- 多实例部署统一使用 Redis，避免本地缓存不一致
- 权限、角色、菜单变更后检查版本键是否刷新
- 排查“权限未生效”时同时检查：缓存键版本 + 用户权限集合 + 协程上下文缓存
- 排查“在线人数异常”时检查 token TTL 与索引过期回收是否正常

## 变更后最小验证清单 ✅

1. 普通用户角色变更后，权限是否立即生效
2. 超管执行全局清理后，权限/在线统计是否刷新
3. 用户登出后，在线会话是否减少
4. 多端登录后，`user_count` 与 `session_count` 是否符合预期
