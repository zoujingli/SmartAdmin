# 新模块开发

新模块应优先复用项目已有的 `CoreController`、`CoreService`、`CoreMapper`、`CoreModel`、权限注解、日志注解、`plugin.json` 应用清单和前端 API 封装。目标是让新模块天然具备分页、软删、数据范围、租户隔离、操作日志、菜单权限和统一响应。

如果模块需要独立发布、独立授权、通过 Composer 接入或携带完整后台/前端能力，请同时阅读 [插件开发](./插件开发.md)。本页聚焦业务模块本身的分层开发，插件打包、授权元数据和安装更新规则以插件开发文档为准。

## 推荐目录

```text
plugin/<Module>/
├── src/
│   ├── Controller/
│   ├── Service/
│   ├── Mapper/
│   ├── Model/
│   ├── Provider.php
│   └── Support/
├── stc/
│   ├── view/
│   ├── languages/
│   └── migrations/
├── composer.json
└── plugin.json
```

建议按业务域拆插件，不要把所有功能继续堆到 `System`。例如客户管理可以是 `plugin/Customer`，财务管理可以是 `plugin/Finance`。跨模块公共能力才上沉到 `SmartAdminLibrary`（仅生态维护者维护），普通业务能力保持独立插件。

## 设计前确认

开发前先回答这些问题：

| 问题 | 决策影响 |
|------|----------|
| 是否需要租户隔离 | 表是否包含 `tenant_id`，Model `$fillable` 是否包含该字段 |
| 是否需要数据范围 | Mapper 是否接入标准数据范围，查询对象用用户字段还是部门字段 |
| 是否需要软删除 | 表是否包含 `deleted_at`，Model 是否使用 `SoftDeletes` |
| 是否需要审计 | Model 是否配置 `$logRules`，写操作是否加 `#[Logger]` |
| 是否需要菜单入口 | 是否在 `plugin.json` 声明应用、菜单、view、路由和按钮权限 |
| 是否进入 release 必要数据 | 是否需要调整 release 配置或种子数据 |

## 后端步骤

1. 创建迁移，明确是否需要 `tenant_id`、软删除、审计字段。
2. 创建 Model，写入 `$fillable`、`$hidden`、关联和转换器。
3. 创建 Mapper，继承 `CoreMapper`，声明 `$model`，实现搜索条件和列表后处理。
4. 创建 Service，继承 `CoreService`，通过构造函数注入 Mapper。
5. 创建 Controller，继承 `CoreController`，为所有保护接口添加 `#[Auth]`。
6. 写操作添加 `#[Logger]`，密码、Token、Secret、Key 等字段必须脱敏。
7. 通过 `plugin.json` 声明应用、菜单、view 和按钮权限。
8. 先执行插件清单 dry-run，再执行菜单和节点同步命令。

如果该模块作为独立插件发布，还需要补充插件 `composer.json`、`Provider.php` 运行期装配、授权协议元数据和发布验证清单。

## 分层职责

| 层 | 负责内容 | 不应承担 |
|----|----------|----------|
| Controller | 路由、参数读取、权限注解、日志注解、标准响应 | SQL、事务、复杂校验 |
| Service | 业务编排、事务、唯一性校验、跨 Mapper 调用 | 请求对象、响应对象、原生 HTTP 输出 |
| Mapper | 查询条件、分页、数据范围、软删、列表后处理 | 业务流程、用户输入直通 SQL |
| Model | 表映射、fillable、hidden、关联、访问器、转换器 | 复杂查询、请求态上下文 |

## Controller 示例

```php
<?php

declare(strict_types=1);

namespace Plugin\Demo\Controller;

use Hyperf\HttpServer\Annotation\Controller;
use Hyperf\HttpServer\Annotation\DeleteMapping;
use Hyperf\HttpServer\Annotation\GetMapping;
use Hyperf\HttpServer\Annotation\PostMapping;
use Hyperf\HttpServer\Annotation\PutMapping;
use Plugin\Demo\Service\DemoService;
use Plugin\Library\Annotation\Auth;
use Plugin\Library\Annotation\Logger;
use Plugin\Library\CoreController;

#[Controller(prefix: '/demo/item')]
#[Auth(name: '演示模块')]
final class DemoController extends CoreController
{
    public function __construct(
        protected readonly DemoService $service
    ) {}

    #[GetMapping(path: 'index')]
    #[Auth(name: '演示列表', type: Auth::CHECK, menu: true, code: 'demo.item.index')]
    public function index(): mixed
    {
        return $this->success($this->service->getPageList($this->request->all()));
    }

    #[PostMapping(path: 'create')]
    #[Auth(name: '新增演示', type: Auth::CHECK, menu: false, code: 'demo.item.create')]
    #[Logger(name: '新增演示')]
    public function create(): mixed
    {
        return $this->success($this->service->create($this->request->all()));
    }

    #[PutMapping(path: 'update/{id}')]
    #[Auth(name: '编辑演示', type: Auth::CHECK, menu: false, code: 'demo.item.update')]
    #[Logger(name: '编辑演示')]
    public function update(int $id): mixed
    {
        return $this->success($this->service->update($id, $this->request->all()));
    }

    #[DeleteMapping(path: 'delete/{ids}')]
    #[Auth(name: '删除演示', type: Auth::CHECK, menu: false, code: 'demo.item.delete')]
    #[Logger(name: '删除演示')]
    public function delete(string $ids): mixed
    {
        return $this->deleteByIds($this->service, $ids);
    }
}
```

示例只展示结构。实际代码应按已有 Controller 的方法命名、返回格式和错误处理实现，不要复制不存在的业务字段。

## Model 和 Mapper 要点

Model 最少要明确：

- 表名和主键策略。
- `$fillable`，禁止请求字段绕过白名单。
- `$hidden`，隐藏密码、Token、Secret、Key 等敏感字段。
- 软删除、时间字段、JSON 转换器和关联关系。
- `$logRules`，为关键业务字段提供中文名、枚举值和单位。

Mapper 最少要明确：

- `protected string $model = XxxModel::class`。
- 查询条件白名单，不允许请求直接控制 `select`、`orderBy`、`raw`、关联名或表达式。
- 列表后处理逻辑，例如关联名称、状态文本、统计字段。
- 数据范围字段，例如创建人字段、用户字段或部门字段。

## Service 要点

Service 应负责业务规则：

- 新增/编辑前做唯一性校验，例如编码、名称、手机号等。
- 多表写入使用事务，例如主表加关系表、发布公告加接收人。
- 关系变更如果不会触发模型字段事件，需要手动追加变更日志。
- 不在单例 Service 属性中保存请求态、用户态、租户态和 Token 用户模型上下文。

## 权限和菜单

新增模块至少需要：

- `plugin.json` 菜单 code：决定前端菜单和按钮展示。
- `#[Auth]` code：决定接口访问控制。
- 前端按钮权限：决定页面操作是否显示。

同步命令在源码/CI 环境执行，用于把插件菜单和权限节点收敛到数据库；插件 ZIP 分发、安装、移除和备份恢复由 SmartAdminLibrary 提供的 `xadmin:plugin:*` 源码命令负责，backup 默认只备份代码，显式 `--with-data` 才包含插件自有表。已发布二进制不提供动态插件安装更新能力，菜单/节点同步也只作为隐藏应急入口保留。

```bash
./bin/smart.php xadmin:menu:sync --dry-run --json
./bin/smart.php xadmin:node:sync --dry-run --json
./bin/smart.php xadmin:menu:sync --details
./bin/smart.php xadmin:node:sync --details
```

## 前端步骤

1. 在插件自己的 `plugin.view_root` 下新增 API service、私有组件和页面，并在 `plugin.json` 中显式配置该目录。
2. 在页面目录内维护表单 schema、表格 columns、常量和类型。
3. 在 `plugin.json` 中声明菜单 route 和相对 `view`。
4. 页面按钮权限码与后端 `#[Auth]` code 保持一致。
5. 新增、编辑表单必须提交后端必需字段。
6. 列表页补齐搜索、分页、状态、回收站、前端导出和统计入口。

独立插件可以把页面放在 `plugin/<Module>/<plugin.view_root>`，但必须显式配置 `plugin.view_root`。菜单只写相对 `view`，后端同步生成 `@plugin/<Module>/views/<path>.vue`，Web 运行库会在编译期根据 `plugin.json` 映射到真实 Vue 页面；插件前端变化后必须重新执行 `composer web:build`，发布打包时 `composer build` 只复用已有 `web/dist`。

## 接口文档要求

每新增一个 Controller，都应在接口参考中补充：

- 控制器路径和接口前缀。
- HTTP 方法、路径、认证类型、权限码、日志名称、用途。
- 列表查询参数。
- 写入字段和关系 payload。
- 权限、数据范围、租户和审计边界。

## 测试建议

- 有数据范围或权限逻辑时，补就近单测。
- 影响公共响应、异常、日志、发布升级时，运行 `composer analyse` 和 `composer test`。
- 影响前端类型或页面时，运行 `composer web:build`。
- 只改 docs 时运行 `composer docs:check`。
- 新增发布种子、菜单或权限时，至少执行菜单和节点同步 dry-run。
- 独立插件还需要检查本地 Composer autoload、`plugin.json` 清单、Provider 装配、授权声明和前后端发布边界。

## 常见错误

- Controller 直接写 SQL，导致数据范围、软删、租户隔离失效。
- 只新增前端按钮，没有新增后端 `#[Auth]` code。
- 菜单 code、按钮 code 和 Controller code 不一致，导致授权后仍然 403。
- Model 漏写 `$fillable`，新增/编辑字段无法保存。
- 支持租户的表漏写 `tenant_id` 或 Model `$fillable` 漏写 `tenant_id`。
- 写操作漏加 `#[Logger]`，生产问题无法追踪。
- 将密码、Token、Secret、Key 写入日志或导出文件。
- 把需要独立发布和授权的业务能力混入 `System`，导致后续安装、更新和维护边界不清。
