# 接口规范

本页定义后端 API 的路径、响应、错误码、认证、日志和前端对接约定。完整接口清单见 [接口参考](../接口参考/README.md)。

## 统一响应结构

```json
{
  "code": 200,
  "info": "操作成功",
  "data": {},
  "path": "/system/user/index"
}
```

项目后端统一返回 `code/info/data/path`。`info` 是标准消息字段，前端不得在不同模块维护多套解析逻辑。

HTTP status 固定返回 `200`；业务状态统一读取 `body.code`，标准业务码只使用 `200/401/403/404/500`。

## 标准 CRUD 路径

| 动作 | 方法 | 路径 |
|------|------|------|
| 列表 | GET | `/{module}/{resource}/index` |
| 回收站 | GET | `/{module}/{resource}/recycle` |
| 详情 | GET | `/{module}/{resource}/info/{id}` |
| 创建 | POST | `/{module}/{resource}/create` |
| 更新 | PUT | `/{module}/{resource}/update/{id}` |
| 删除 | DELETE | `/{module}/{resource}/delete/{ids}` |
| 彻底删除 | DELETE | `/{module}/{resource}/real-delete/{ids}` |
| 恢复 | PUT | `/{module}/{resource}/recovery/{ids}` |
| 状态 | PUT | `/{module}/{resource}/status/{id}` |

特殊接口，例如认证、上传、公告收件箱、系统看板，可以在标准路径外单独声明，但仍要遵守响应、权限、日志和数据范围约束。

## 错误码约定

| code | 含义 |
|------|------|
| 200 | 成功 |
| 401 | Token 缺失、过期、无效或刷新失败 |
| 403 | Token 有效但无操作权限 |
| 404 | 页面或 API 路由不存在 |
| 500 | 业务异常、校验失败、数据不存在、账号密码错误、账号禁用或系统异常 |

账号密码登录失败属于业务失败，返回 `500`；只有依赖当前 Token 的接口在认证失败时才返回 `401`。

## 认证入口落地示例

System 后台和业务插件应用的认证入口固定分离，禁止在同一个接口中混用用户模型。下表以当前内置 Project 插件为落地示例：

| 入口 | 登录接口 | 用户模型 | 业务接口范围 |
|------|----------|----------|--------------|
| System 后台 | `POST /system/auth/login` | `SystemUser` | `/system/**`、`/system/project/**` |
| Project 应用 | `POST /project/account/auth/login` | `ProjectAccount` | `/project/**` |

System `AuthController` 认证链路包含：

- `POST /system/auth/login`：用户名密码登录，校验账号状态，更新最后登录信息。
- `POST /system/auth/refresh`：刷新 Token，失败返回 `401`。
- `POST /system/auth/profile`：返回前端会话所需完整用户资料。
- `GET /system/auth/codes`：返回当前用户权限码集合，超级管理员返回 `*`。
- `POST /system/auth/logout`：退出登录。

业务插件账号认证链路应由插件自己的认证控制器提供；以 Project 为例，资料和权限码返回 `auth_user_model=Plugin\Project\Model\ProjectAccount`。两类 Token 不能跨入口访问，跨用时按未登录返回。

界面初始化存在两个元信息接口：`/system/auth/ui-meta` 是公开接口，供登录页和应用启动读取低敏展示配置；`/system/data/ui-meta` 是登录态兼容接口，接受实现 `UserModelInterface` 的有效 Token，例如 `SystemUser` 或 `ProjectAccount`，只返回低敏 UI 元信息。这个兼容边界不能扩展到业务数据、个人资料、权限码或系统配置接口。

这部分接口体现项目原则：Controller 负责协议层，用户信息组装和业务判断下沉到 Service。

## 接口安全要求

- 受保护接口必须配置 `#[Auth]`；业务插件前台接口必须显式写入对应用户模型，例如 `userModel: ProjectAccount::class`。
- 写操作必须配置 `#[Logger]`。
- 请求体敏感字段不得原样写入日志。
- 查询参数需要白名单过滤，禁止请求直接控制 `select`、`orderBy`、raw SQL、关联名或字段名。
- 非标准接口，例如工作台、统计、用户选项，同样必须遵守权限和数据范围；前端导出应复用标准列表接口。

## 日志与审计约束

关键接口应配合 `#[Logger]`：

- 登录接口排除 `password`。
- 修改密码排除 `old_password/new_password`。
- 上传配置、密钥、Token、Secret、Key、Cookie、Authorization 等字段必须脱敏。
- 列表和详情的日志响应数据只记录预览，超出长度限制后截断并脱敏。

`LogsActionService` 负责操作日志落库、统计与实时窗口分析；`LogsChangeService` 负责业务变更明细查询。接口设计应与日志查询维度保持一致，例如业务名、响应码、用户、业务模型、时间范围。
