# 编码规范

## 通用要求

- PHP 文件必须启用 `declare(strict_types=1);`。
- 新建业务类优先使用 `final class`。
- 通过构造函数注入依赖，避免服务定位器式散落调用。
- 主流程优先线性可读，不为少量一次性逻辑过度拆分私有方法。
- Swoole 环境不要在单例 Service 属性中保存请求态、用户态、租户态或 Token 用户模型上下文。

## 分层规则

| 层 | 规则 |
|----|------|
| Controller | 只做路由、参数读取、权限注解、日志注解和标准响应 |
| Service | 处理业务编排、事务、唯一性校验、跨 Mapper 调用和领域规则 |
| Mapper | 处理数据访问、查询条件、数据范围、分页、软删、状态和列表后处理 |
| Model | 处理表映射、fillable、hidden、关联、访问器和转换器 |

## 数据库命名规范

- 插件表名必须以模块名转换后的下划线前缀开头，例如 `WechatClient` 使用 `wechat_client_`，`WechatService` 使用 `wechat_service_`。
- Model 必须显式声明 `$table`，不要依赖基类按类名自动推导，避免类名简化或业务缩写调整时访问到错误表。
- 支付业务统一使用完整 `payment` 命名，例如 `wechat_client_payment_order`、`WechatClientPaymentOrderService`，不要使用 `pay` 缩写。
- 公众号粉丝在本地数据实体中统一使用 `user` 命名，例如 `wechat_client_user`、`WechatClientUserService`；后台菜单与业务文案仍可展示“粉丝管理”。
- 微信开放平台授权账号、回调日志、网关凭据按实体语义分别使用 `auth`、`logger`、`gateway`，字段可以继续保留微信官方术语，如 `authorizer_appid`。
- 新增或调整迁移时，同步更新文件名、`Schema::create/dropIfExists`、索引名、表注释、Model、Mapper、Service、测试和文档。

## 注解与日志

- 受保护接口必须声明 `#[Auth]`。
- 写操作必须声明 `#[Logger]`。
- 敏感字段必须配置脱敏，避免密码、Token、Secret、Key、Cookie、Authorization 入库或入日志。
- 关键业务 Model 如需审计，应定义 `$logRules`。

## 文档同步

发生以下变化时必须同步文档：

- 路由、接口参数或响应结构变化。
- 权限节点、菜单基线或按钮权限变化。
- 租户、数据范围、日志、上传、缓存、发布机制变化。
- 前端启动、构建、部署或运行时配置变化。
