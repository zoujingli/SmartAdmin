# 插件开发

SmartAdmin 的业务能力优先通过 `plugin/<Module>` 扩展。插件是当前仓库内的本地模块，通过本地 Composer path 包提供 autoload 与 Hyperf Provider 装配，前端资源在 Vue/Vite 构建期打包进 `web/dist`。

当前标准机制是“本地 Composer path 包 + Provider 装配 + `plugin.json` 业务清单 + 菜单/节点同步 + Web 编译期扫描”。它不是运行时远程插件加载，也不再通过 Composer 在发布包内安装、更新或删除插件；如需 ZIP 分发、安装、移除、备份和恢复，只能在源码/CI 模式使用 Library 内置 `xadmin:plugin:*` 命令。插件页面、路由或认证入口变更后需要重新生成前端产物。

本文统一采用以下口径：

- Composer 只负责本地 path 包 autoload 与 Provider 配置；不要把后台应用、菜单、view、按钮权限或授权展示信息塞进 Composer 元数据。
- `Provider.php` 只负责运行期装配，例如注解扫描、依赖绑定、监听器和命令；不要用 Provider 注册菜单、模块目录或前端页面。
- `plugin.json` 是应用、菜单、按钮权限、模块摘要、授权展示、`view_root`、`language_root`、`migration_root` 的唯一推荐业务清单。
- Library 内置 `xadmin:plugin:*` 是源码/CI 插件 ZIP 和备份工具，生产 Phar/SFX 不注册这些命令。
- 菜单同步、节点同步只在源码/CI 环境作为构建和校验命令使用；已发布 Phar/SFX 二进制不支持在包内动态安装、更新或移除插件。
- `composer build` 只复用并校验已有 `web/dist`；插件前端变化必须先执行 `composer web:build` 或由 CI 提供等价前端产物。

## 插件边界

| 类型 | 说明 | 是否普通业务插件 |
|------|------|------------------|
| `plugin/Library` | Core 基类、注解、切面、中间件、异常、日志、数据范围、租户上下文 | 否 |
| `plugin/System` | 登录、权限、菜单、组织、文件、日志、公告、系统数据等后台基础能力 | 否 |
| `plugin/Builder` | Phar 打包、发布构建、配置改写和二进制产物生成 | 否 |
| `plugin/<Business>` | 客户、财务、工单、资产、CRM 等业务域能力 | 是 |

内置 `Library/System/Builder` 是基础插件，不按普通业务插件卸载。新业务功能应优先作为独立业务插件接入，不要继续堆入 `System`。

## 标准接入链路

插件接入按以下顺序维护，避免本地包、业务清单、数据库菜单和前端产物互相漂移：

1. 在源码中创建或更新 `plugin/<Module>`，并通过根 `composer.json` 的 path repository / require 接入本地包。
2. 插件 `Provider.php` 只声明运行期装配能力，例如注解扫描、依赖绑定、监听器和命令。
3. 插件 `plugin.json` 声明应用、菜单、按钮权限、模块摘要、授权展示以及 `view_root/language_root/migration_root` 等资源根。
4. 如需 ZIP 分发或环境间迁移，在源码/CI 环境通过 Library 内置 `xadmin:plugin:package/install/remove/backup/restore` 处理插件文件、根 Composer 依赖和插件自有表快照。
5. 在源码/CI 环境执行 `xadmin:menu:sync`、`xadmin:node:sync` dry-run，将菜单和权限节点同步口径确认到系统表。
6. 执行 `composer web:build`，Web 壳在编译期扫描已配置 `plugin.view_root` 的插件页面、`routes.ts` 和 `auth-entry.ts`。
7. 发布前执行文档、静态检查、菜单节点 dry-run 和发布检查，确保清单和构建产物一致。

## 目录规范

当前推荐的应用插件结构：

```text
plugin/<Module>/
├── src/                  # 插件运行期 PHP 代码
│   ├── Command/          # 可选，插件命令
│   ├── Controller/       # HTTP Controller
│   ├── Event/            # 可选，事件和监听器
│   ├── Mapper/           # 数据访问
│   ├── Model/            # 模型映射
│   ├── Service/          # 业务编排
│   ├── Support/          # 插件内部支持类
│   └── Provider.php      # Hyperf 装配入口
├── stc/                  # 插件静态资料和开发期资源
│   ├── view/             # 前端 view 根目录，由 plugin.view_root 显式启用
│   │   └── auth-entry.ts # 可选，插件前台独立认证入口声明
│   ├── languages/        # 插件语言包，由 plugin.language_root 显式启用
│   └── migrations/       # 插件开发期迁移，由 plugin.migration_root 显式启用
├── composer.json         # 本地 Composer path 包、autoload 和 Provider
├── plugin.json           # SmartAdmin 应用、菜单、view、按钮权限和模块元数据
└── readme.md
```

`web/apps/web-antd` 是通用前端壳、公共页面、共享组件和编译期插件宿主。插件页面由插件自己管理，必须在 `plugin.json` 的 `plugin.view_root` 显式指定插件目录内相对路径后才会参与构建映射。新增插件通常不需要修改 Web 页面、路由或构建配置，只需重启 Vite dev 或重新 build。

插件业务页面可以复用 Web 公共组件，但不能把具体业务页面下沉到 Web 壳。列表页搜索/筛选区统一导入 `#/components/crud-search-field.vue`：

- `Input`、`Select`、`DatePicker`、`RangePicker`、`InputNumber`、`TreeSelect`、`Cascader` 等搜索控件统一放入 `<SearchField label="四字标签">`。
- `label` 必须固定 4 个汉字，优先使用 `搜索内容`、`当前状态`、`所属产品`、`所属版本`、`日期范围`、`创建时间`、`接口账号` 等标准文案。
- 搜索区布局使用 `<Row class="crud-search-grid">`，普通输入、下拉、日期、日期范围统一走 `--crud-search-item-width` 宽度机制；不要因为日期区间、关键字或按钮区再手写不同的超长列。
- 下拉提示宽度由公共组件统一处理，最小等于输入框宽度，按选项内容自动加宽，最大不超过 2 倍输入框宽度；业务页面不要单独写 `popupClassName` 或固定超宽 `dropdownStyle`。
- 查询、重置等按钮放在最后一个 `<Col class="crud-search-grid__actions">`，按钮不使用 `block`，不拉满搜索区剩余宽度。
- 搜索区只负责筛选条件的统一视觉；新增、编辑、详情和业务配置表单继续使用 `FormItem label + 控件`，不要把 `SearchField` 当作通用表单项。
- 存量插件和新增插件都按同一口径执行；代码评审时必须确认所有 `SearchField` 所在行使用 `crud-search-grid`，动作按钮使用 `crud-search-grid__actions`，标签为 4 个汉字。
- 插件页面新增搜索区后必须纳入 `composer web:build`，确保公共组件、插件 view 扫描和主题样式一起通过构建。

插件列表页右侧操作列统一导入 `#/components/crud-table-actions.vue`：

- 操作列仍使用 `fixed: 'right'`，但 `width` 要通过 `estimateVisibleActionColumnWidth()` 估算，不能为了多个按钮直接写超大固定宽度。
- `actions` 数组按业务优先级传入 `label / visible / danger / disabled / confirmTitle / confirmContent / onClick`；`visible=false` 的按钮不参与展示和宽度计算。
- 可见操作数不超过 3 个时平行展示；超过 3 个时自动显示前 2 个 + `更多` 下拉，删除、彻底删除、取消等危险操作放在数组后部。
- 行状态会影响按钮数量时，优先用当前页 `items.map(rowActions)` 估算操作列宽；只有权限固定、按钮固定的页面才用 `[rowActions({})]`。
- 删除、彻底删除、发布、同步等高风险操作必须保留二次确认；如果业务函数内部已经 `Modal.confirm`，组件 action 不要再重复配置 `confirmTitle`。

## Composer 配置

插件仍通过根 `composer.json` 的 path repository 作为本地包接入，用于 autoload 与 Hyperf Provider 装配；插件源码固定保留在 `plugin/<Module>`，不通过 Composer 发布包安装到运行环境：

```json
{
  "repositories": [
    {
      "type": "path",
      "url": "plugin/Demo"
    }
  ],
  "require": {
    "vendor/smart-admin-plugin-demo": "^1.0"
  }
}
```

插件自身 `composer.json` 只负责 Composer 维度：包名、`type=library`、autoload、Hyperf Provider 和授权协议。应用、菜单、view、按钮权限和授权展示信息统一写入 `plugin.json`：

```json
{
  "name": "vendor/smart-admin-plugin-demo",
  "description": "SmartAdmin demo plugin",
  "type": "library",
  "license": "Apache-2.0",
  "autoload": {
    "psr-4": {
      "Plugin\\Demo\\": "src/"
    }
  },
  "extra": {
    "hyperf": {
      "config": "Plugin\\Demo\\Provider"
    }
  }
}
```

`extra.hyperf.config` 会通过 Composer lock 合并到 Hyperf Provider 配置中。新增或修改 Provider、autoload 或依赖后，应重新执行 Composer install/update，确认 Provider 生效并重启 Hyperf 进程。

## plugin.json 应用清单

`plugin.json` 是 SmartAdmin 应用插件的业务清单，后台应用管理、菜单同步、模块矩阵和 Web view 自动加载都以它为主来源：

```json
{
  "plugin": {
    "code": "demo",
    "name": "演示插件",
    "version": "1.0.0",
    "description": "演示插件接入规范",
    "view_root": "stc/view",
    "language_root": "stc/languages",
    "migration_root": "stc/migrations"
  },
  "apps": [
    {
      "id": 9000,
      "name": "演示应用",
      "route": "/demo",
      "component": "BasicLayout",
      "redirect": "/demo/order",
      "icon": "lucide:boxes",
      "sort": 90,
      "type": "D",
      "menus": [
        {
          "id": 9010,
          "name": "订单管理",
          "code": "demo.order.index",
          "route": "/demo/order",
          "view": "order/index.vue",
          "icon": "lucide:list",
          "sort": 10,
          "module": {
            "key": "demo-order",
            "summary": "演示订单管理入口。",
            "features": ["列表检索", "状态管理", "前端导出"]
          },
          "permissions": [
            { "id": 90101, "name": "新增订单", "code": "demo.order.create", "sort": 10 },
            { "id": 90102, "name": "编辑订单", "code": "demo.order.update", "sort": 20 },
            { "id": 90103, "name": "删除订单", "code": "demo.order.delete", "sort": 30 }
          ]
        }
      ]
    }
  ]
}
```

字段约束：

| 字段 | 说明 |
|------|------|
| `plugin.view_root` | 插件自管理 view 根目录，必须是插件目录内相对路径；未配置时菜单 `view` 不生效 |
| `plugin.language_root` | 插件语言包目录，必须是插件目录内相对路径；未配置时不加载插件语言包 |
| `plugin.migration_root` | 插件数据库迁移目录，必须是插件目录内相对路径；未配置时不注册插件迁移 |
| `plugin.tables` | 可选，插件自有表清单；插件备份、恢复、移除只处理这些显式表和前缀命中的表 |
| `plugin.table_prefixes` | 可选，插件自有表前缀；未配置 `tables/table_prefixes` 时按 `plugin.code` 转下划线前缀兜底 |
| `apps[]` | 应用根菜单，通常 `component` 为 `BasicLayout` |
| `menus[].view` | 相对 `plugin.view_root` 的 Vue 文件，如 `order/index.vue` |
| `menus[].module` | 模块矩阵展示信息，可选 |
| `permissions[]` | 按钮权限节点，只声明 `id/name/code/sort` 等基础字段 |

同步时后端会把 `view` 统一转换为稳定组件别名 `@plugin/<Module>/views/<view>`，因此数据库和菜单不需要感知真实 `view_root`。`id`、`code`、`route` 会做全局唯一性校验，`view` 会校验文件存在；路径禁止绝对路径和 `..`。

## 授权协议

SmartAdmin 主项目使用 Apache-2.0，不代表所有第三方插件都自动采用 Apache-2.0。插件以自身 `LICENSE`、`composer.json` 的 `license` 字段和发布方说明为准。

插件 `composer.json` 必须声明 `license`，可使用：

| 值 | 适用场景 |
|----|----------|
| `Apache-2.0` | Apache 2.0 开源插件 |
| `MIT` | MIT 开源插件 |
| `proprietary` | 闭源、会员、付费或私有插件 |
| 自定义标识 | 发布方自定义商业授权 |

建议在 `plugin.json` 的 `plugin` 对象中补充授权展示元数据：

| 字段 | 说明 |
|------|------|
| `license_name` | 授权名称，例如 `Apache License 2.0`、`会员授权`、`商业授权` |
| `license_type` | 授权类型，建议值为 `open_source`、`member`、`paid`、`commercial`、`private` |
| `license_url` | 授权说明页面或 LICENSE 文件地址 |
| `pricing_type` | 价格类型，例如 `free`、`member`、`paid`、`custom` |
| `support_url` | 技术支持、购买或会员入口 |

当前阶段只展示授权信息，不做联网权益校验。付费、会员和私有插件应由源码仓库、授权下载地址或人工交付控制代码分发。

## Provider 接入

插件 `Provider.php` 只负责把运行期能力装配给 SmartAdmin，例如注解扫描、依赖绑定、事件监听和命令。菜单、应用、模块目录和 view 路径必须由 `plugin.json` 声明。

```php
<?php

declare(strict_types=1);

namespace Plugin\Demo;

final class Provider
{
    public function __invoke(): array
    {
        return [
            'annotations' => [
                'scan' => [
                    'paths' => [__DIR__],
                ],
            ],
            'dependencies' => [
                // Interface::class => Implementation::class,
            ],
            'listeners' => [
                // Listener::class,
            ],
            'commands' => [
                // DemoCommand::class,
            ],
        ];
    }
}
```

Provider 只声明运行期装配信息，不保存请求态、用户态、租户态或 Token 用户模型上下文。注解扫描路径应限制在插件 `src` 目录，避免扫描无关目录。菜单、模块根和模块摘要必须写入 `plugin.json`。

## 后端开发规范

插件后端仍按项目标准分层：

| 层 | 要求 |
|----|------|
| Controller | `final class XxxController extends CoreController`，声明 `#[Auth]`，写操作声明 `#[Logger]` |
| Service | `final class XxxService extends CoreService`，负责事务、唯一性校验、跨 Mapper 编排 |
| Mapper | `final class XxxMapper extends CoreMapper`，声明 `$model`，统一处理分页、搜索、软删和数据范围 |
| Model | `final class Xxx extends CoreModel`，声明 `$fillable`、`$hidden`、转换器、关联和 `$logRules` |

关键约束：

- 支持租户的业务表必须包含 `tenant_id`，Model `$fillable` 必须包含该字段。
- 支持软删除的表必须包含 `deleted_at`，Model 使用 `SoftDeletes`。
- 需要审计的模型配置 `$logRules`；关系表、批量授权等模型事件不稳定场景由 Service 手动追加变更日志。
- Controller 不写 SQL，不把 Request/Response 传入 Mapper。
- 权限码、菜单 code、前端按钮 code 必须保持一致。

## 后端语言包

插件语言包放在对应插件目录，并通过 `plugin.json` 的 `plugin.language_root` 显式启用，不把业务文案集中堆到 `storage/languages`：

```text
plugin/<Module>/stc/languages/
├── zh_CN/<group>.php
├── zh_TW/<group>.php
└── en_US/<group>.php
```

约定：

- `plugin/Library/languages` 只放通用响应、异常、基础设施文案，例如 `操作成功`、`数据不存在`。
- `plugin/System/stc/languages` 只放 System 插件业务文案，例如用户、角色、菜单、文件、字典等错误提示。
- 业务插件使用自己的 group，例如 `plugin/Crm/stc/languages/zh_CN/crm.php`，调用 `__('crm.客户不存在')`。
- 后端业务异常可以直接抛中文文案；公共异常会优先查找 `library`、`system`、`plugin`、`builder`，再扫描已配置 `plugin.language_root` 的插件语言文件名作为 group，缺失时显示原中文。
- `storage/languages` 仅作为项目级覆盖目录使用，不再放插件默认语言包。

## 菜单和权限

插件菜单、应用根、页面 view、模块摘要和按钮权限必须写入 `plugin.json`，由 `PluginManifestRegistry` 统一读取并交给菜单同步服务导入。如果不同插件清单中出现重复菜单 ID、权限 code 或 route，系统会 fail-fast，避免不确定覆盖。

权限节点来源分两类：

| 来源 | 说明 |
|------|------|
| 注解节点 | 后台 `SystemUser` Controller `#[Auth]` 同步到 `system_node`，用于 System RBAC 接口权限校验 |
| 菜单节点 | `plugin.json` 菜单和按钮 code 同步到菜单树，用于前端菜单、按钮和角色授权 |

如果插件需要提供非后台用户体系，例如业务插件自有的 `DemoAccount`，应使用独立路径和显式 `#[Auth(userModel: Xxx::class)]` 隔离；这类前台权限不要进入 System RBAC，也不要写入后台角色授权链路。

独立前台入口的权限码可以和后台管理入口同名，但来源必须分离。例如业务插件的 `/system/demo/**` 使用 System RBAC，`/demo/**` 使用插件自有账号和角色矩阵，两边都可以使用 `demo.order.index` 表达同一业务动作，但不能互相授权或复用 Token。

同步命令：

```bash
sh bin/start-swoole xadmin:menu:sync --dry-run --json
sh bin/start-swoole xadmin:node:sync --dry-run --json
sh bin/start-swoole xadmin:menu:sync --details
sh bin/start-swoole xadmin:node:sync --details
```

新增插件后先执行菜单和权限节点 dry-run 校验 `plugin.json`，确认无误后执行正式同步。

## 前端接入

Web 目录只作为通用壳、布局、公共组件、共享依赖和插件扫描消费层。业务插件的 API service、私有组件和页面应放在插件自己配置的 `plugin.view_root` 目录下，并通过相对路径组织。

命名建议：

| 内容 | 约定 |
|------|------|
| API 文件 | `plugin/<Module>/<view_root>/api/<resource>.ts` 或页面就近 service |
| 页面目录 | `plugin/<Module>/<view_root>/<resource>/` |
| 类型 | `DemoInfo`、`DemoFormData`、`DemoOption` |
| 权限码 | `<module>.<resource>.<action>` |
| 菜单路由 | `/<module>/<resource>` |

插件前端资源通过 `plugin.json` 自动加载：

```text
plugin/Demo/plugin.json        # plugin.view_root = "stc/view"
plugin/Demo/stc/view/order/index.vue
```

`plugin.json` 中菜单只需要写：

```json
{ "route": "/demo/order", "view": "order/index.vue" }
```

后端同步时会生成 `@plugin/Demo/views/order/index.vue`；Web 运行库会在编译期读取已配置的 `plugin.view_root` 并自动把该别名映射到真实 Vue 页面。插件内页面如果依赖同插件的 API 或组件，建议使用相对路径导入；跨插件稳定复用的 UI 再沉淀到 Web 公共组件或公共包。

### 插件认证入口

如果插件需要提供独立前台登录页，应在自己的 `plugin.view_root` 下声明 `auth-entry.ts` 或 `auth-entry.mts`。Web 壳会在编译期扫描所有已配置 `plugin.view_root` 的插件并读取这些入口配置，不允许为了某个插件登录页去修改 `web/apps/web-antd` 的固定路由或硬编码插件名。

最小示例：

```ts
export default {
  authBase: '/demo/account/auth',
  entry: 'demo',
  homePath: '/demo/order',
  loginPath: '/demo/login',
  name: '演示应用',
  permissionPrefixes: ['demo.'],
  routePrefixes: ['/demo'],
  userModel: 'Plugin\\Demo\\Model\\DemoAccount',
  userModelIncludes: ['DemoAccount'],
};
```

字段约束：

| 字段 | 说明 |
|------|------|
| `entry` | 入口唯一标识，不能使用内置 `system` |
| `authBase` | 插件账号认证控制器前缀，需提供 `login/refresh/profile/codes/logout/password-crypto` |
| `loginPath` | 插件登录页路由，例如 `/demo/login` |
| `homePath` | 登录成功后的默认页面，必须位于 `routePrefixes` 范围内 |
| `routePrefixes` | 插件前台路由边界，路由守卫按它判断当前入口 |
| `permissionPrefixes` | 插件前台权限码前缀，用于裁剪插件菜单和动态路由 |
| `userModel` / `userModelIncludes` | 用于识别登录资料中的 `auth_user_model` 是否属于当前入口 |
| `menus` | 可选，插件前台菜单树；System 后台菜单仍由 `plugin.json` 同步 |
| `profilePath` / `profile` | 可选，插件个人资料页路径和展示文案 |

后端配套要求：

- 插件认证控制器的登录返回必须包含 `token`、`user` 和 `auth_user_model`。
- 插件 `profile` 接口必须返回同一个用户模型标识，前端据此区分入口。
- 插件业务接口必须显式声明 `#[Auth(userModel: Xxx::class)]`，不要依赖默认 `SystemUser`。
- 共享初始化元信息应走 `/system/auth/ui-meta`；已登录后的兼容元信息可走 `/system/data/ui-meta`，但该接口只能返回低敏 UI 配置。

## 本地接入和变更动作

插件新增或升级的标准动作仍是合并源码、更新本地包元数据、同步菜单节点并重新构建前端产物；如需交付 ZIP，可在源码/CI 模式使用 Library 内置 `xadmin:plugin:*` 命令打包、安装、移除和备份恢复。发布 Phar/SFX 二进制不注册这些命令。

### Library 内置源码插件命令

这些命令只在源码/CI 模式出现，命令类必须使用 `SourceOnlyCommand`；生产 Phar/SFX 不注册 `xadmin:plugin:*`。

```bash
php bin/hyperf.php xadmin:plugin:package <插件名或目录> -o runtime/plugin/packages -p <zip密码>
php bin/hyperf.php xadmin:plugin:install <插件zip路径或URL> -p <zip密码> [--force]
php bin/hyperf.php xadmin:plugin:backup <插件名或目录> [--with-data] -p <zip密码>
php bin/hyperf.php xadmin:plugin:remove <插件名或目录> -p <备份zip密码>
php bin/hyperf.php xadmin:plugin:restore <备份zip路径、URL或默认备份名称> -p <zip密码> [--force] [--no-data]
```

普通插件包 ZIP 顶层为 `composer.json`、`plugin.json`、`src/`、`stc/` 等；备份包默认包含插件代码和 `_xadmin/plugin-backup.json`，`plugin-backup.json` 会记录 `with_data`、插件版本、表清单和行数摘要。只有 `backup --with-data` 或 `remove` 自动备份才额外包含 `_xadmin/database.schema.gz` 与 `_xadmin/database.data.gz`；带数据备份恢复时默认恢复数据，可用 `restore --no-data` 只恢复代码。`-p/--password` 使用 ZIP AES-256 加密，安装/恢复加密包时必须提供密码。安装和恢复会维护根 `composer.json` path repository / require 并执行 Composer 更新；若含前端资源，仍需执行 `composer web:build`。

备份默认写入 `runtime/plugin/backups`，文件名为 `插件code-版本号-backup-YYYYMMDD-HHMMSS.zip`，例如 `project-1.0.0-backup-20260522-123000.zip`。恢复参数仅传文件名时从默认备份目录读取，且 `.zip` 可省略；传绝对路径或带目录的相对路径时按指定路径读取。

`Library/System/Builder` 是基础插件，不允许通过这些命令移除或覆盖。插件数据范围只按 `plugin.tables`、`plugin.table_prefixes` 或 `plugin.code` 推导前缀处理，不读取 release 的全局 `backup_tables`。

### 变更类型和必需动作

| 变更 | 必需动作 |
|------|----------|
| 新增插件目录或修改 `composer.json`、Provider、autoload | 更新根 path repository / require，重新 Composer install/update，确认 Provider 生效并重启 Hyperf |
| 修改 `plugin.json` 的应用、菜单、按钮、资源根或授权展示 | 菜单/节点 dry-run，确认后正式同步 |
| 修改 Controller `#[Auth]` code | 节点同步 dry-run，确认角色授权树和前端按钮权限一致 |
| 修改插件页面、`routes.ts`、`auth-entry.ts` 或 API service | `composer web:build` 重新生成 `web/dist`；发布时再执行 `composer build` |
| 修改语言包或迁移目录 | 确认 `plugin.language_root` / `plugin.migration_root` 已显式配置，并运行对应后端检查 |
| 发布二进制 | CI/人工先提供 `web/dist`，再执行 `composer build`；生产只运行已打包插件和运行期命令 |

常用校验命令：

```bash
composer validate
sh bin/start-swoole xadmin:menu:sync --dry-run --json
sh bin/start-swoole xadmin:node:sync --dry-run --json
composer web:build
```

## 发布升级

插件发布时需要明确数据归属：

- 迁移用于开发期 fresh 初始化，表名、字段、索引和注释必须完整。
- 发布升级依赖 DBAL 快照机制，不把生产升级绑定到逐条 migration。
- 受插件管理的基线菜单、权限节点和模块摘要应通过 `plugin.json` 与同步服务处理；字典、初始化数据按业务种子或快照策略处理。
- 运行数据不要放入发布强接管数据，日志、上传文件、业务流水等应按发布规则排除或备份。
- Phar 构建前确认插件源码、Provider、`plugin.json`、迁移和前端 view 都进入构建流程；插件前端产物应先由 `composer web:build` 或 CI 生成，`composer build` 只复用并打包已有 `web/dist`。
- 已发布二进制只保留运行、升级和必要维护命令；插件新增或升级必须重新构建前端产物和发布包。

## 验证清单

后端：

- 新增或修改的 PHP 文件通过语法检查。
- 涉及公共逻辑时执行 `composer analyse` 和 `composer test`。
- 菜单和权限节点执行 dry-run，再正式同步。
- 登录普通角色验证菜单、按钮和接口权限。
- 数据范围、租户隔离、软删除、恢复、彻底删除都按预期生效。

前端：

- API service、页面、路由、菜单 component 和按钮权限码一致。
- 列表、搜索、分页、新增、编辑、删除、恢复、导出和状态切换可用。
- 执行 `composer web:build` 或在 WSL 中执行对应 pnpm typecheck/build。

文档：

- 更新用户教程、系统功能、接口参考和部署说明。
- 执行 `composer docs:check`。

## 常见问题

| 问题 | 排查方向 |
|------|----------|
| 插件 Controller 无法访问 | 检查 `extra.hyperf.config`、Provider 注解扫描路径、Composer lock/install/update 和进程重启 |
| 权限节点缺失 | 检查 Controller `#[Auth]`、节点同步命令和同步输出 |
| 菜单显示但页面空白 | 检查 `plugin.json` 的 `plugin.view_root`、菜单 `view` 和真实 Vue 文件路径 |
| 按钮显示但接口 403 | 检查按钮 code、角色授权和后端 `#[Auth]` code |
| 插件登录后被踢回登录页 | 检查 `auth-entry.ts` 的 `loginPath`、`authBase`、`routePrefixes`、`userModel`，以及后端 `profile` 是否返回正确 `auth_user_model` |
| 插件权限码接口提示未登录 | 检查业务 Controller 是否显式声明插件用户模型，前端当前入口是否和登录页入口一致 |
