# 前端接入

前端主应用位于 `web/apps/web-antd`，使用 Vue 3、Vite、Ant Design Vue 和 Vben Admin 体系。

## 目录结构

```text
web/apps/web-antd/src/
├── api/                 # API service、类型和请求封装
├── router/              # 路由配置
├── store/               # 登录态、用户态、权限态
├── views/               # 页面
└── utils/               # 通用工具
```

Web 核心页面可以放在 `src/views/<module>/<resource>`，接口封装放在 `src/api/<module>/<resource>.ts`。业务插件页面不要放入 Web 壳，应放在插件自己的 `plugin.view_root` 下；页面内复杂表单、表格列、常量可以拆到同目录的 `data.ts`、`constants.ts`。

如果前端页面属于独立业务插件，请同时阅读 [插件开发](./插件开发.md)。Web 只提供通用壳、运行库、公共组件和插件扫描消费逻辑；插件页面由 `plugin/<Module>/plugin.json` 的 `plugin.view_root` 显式声明后参与编译期映射，插件独立登录入口由插件 view 下的 `auth-entry.ts` 声明。

## 接入步骤

1. Web 核心功能在 `src/api/<module>` 增加 API service，插件功能在插件 `plugin.view_root` 下就近维护 API service。
2. Web 核心功能在 `src/views/<module>` 增加页面，插件功能在 `plugin/<Module>/<view_root>` 增加页面、表单和类型定义。
3. Web 核心静态页面在 `src/router/routes/modules` 注册路由；插件公开路由放在插件 `routes.ts`，后台菜单路由通过 `plugin.json` 生成动态路由。
4. 页面按钮权限与后端 `#[Auth]` code 保持一致。
5. 表单提交字段必须覆盖后端必填字段，不能依赖危险默认值。

## API service

现有模块通常按资源维护 service，例如：

- `src/api/system/user.ts`
- `src/api/system/role.ts`
- `src/api/system/menu.ts`
- `src/api/system/file.ts`
- `src/api/system/logs-action.ts`
- `src/api/system/logs-change.ts`
- `src/api/system/notice.ts`
- `src/api/system/tenant.ts`

新增 service 时建议保持以下约定：

| 方法 | 说明 |
|------|------|
| `getXxxList` | 正常列表 |
| `getXxxRecycleList` | 回收站列表 |
| `getXxxDetail` | 详情 |
| `createXxx` | 新增 |
| `updateXxx` | 编辑 |
| `deleteXxx` / `batchDeleteXxx` | 软删除 |
| `recoveryXxx` | 恢复 |
| `realDeleteXxx` | 彻底删除 |
| `updateXxxStatus` | 状态切换 |
| `getXxxStatistics` | 统计 |
| 前端 `handleExport` | 复用当前筛选条件和列表分页接口生成 xlsx |
| `getXxxOptions` | 下拉选项 |

接口返回统一走项目请求封装处理，页面层不要绕过统一拦截器直接解析异常结构。

## API 地址

前端通过运行时配置和环境变量确定 API 地址：

- 开发环境可直连后端，也可使用 Vite 代理。
- 构建产物通过 `_app.config.js` 注入 `VITE_GLOB_API_URL`。
- 后端 `SiteMiddleware` 会动态生成 `_app.config.js`。

## 页面形态

标准管理页建议包含：

| 区域 | 内容 |
|------|------|
| 搜索区 | 关键字、状态、时间范围、组织或业务筛选 |
| 工具栏 | 新增、批量删除、导出、刷新、列设置 |
| 表格 | 分页、排序、状态展示、操作列 |
| 回收站 | 软删除数据、恢复、彻底删除 |
| 统计区 | 总量、启用/禁用、近期变化、业务指标 |
| 表单弹窗 | 新增、编辑、详情、关系分配 |

用户、角色、文件、日志、公告、租户页面已经实现这些模式。新模块应优先复用这些页面结构，避免每个模块都自定义交互。

## 搜索区标准

所有列表页、报表页、日志页和插件业务页的搜索/筛选区必须使用统一搜索控件 `#/components/crud-search-field.vue`，把筛选标签固定放在控件前缀视觉位，避免不同页面的 `Input`、`Select`、`DatePicker`、`RangePicker`、`InputNumber` 等组件高度、边框和间距不一致。

使用规则：

- 搜索区每一个筛选控件都用 `<SearchField label="四字标签">...</SearchField>` 包裹；搜索、重置、新增、导出等按钮不包裹。
- `label` 固定 4 个汉字，常用文案包括：`搜索内容`、`当前状态`、`所属产品`、`所属版本`、`日期范围`、`创建时间`、`操作用户`、`响应代码`、`接口账号`。
- 控件内部保持原查询参数、事件和远程搜索逻辑不变，`Select / RangePicker / InputNumber` 等需要继续加 `class="w-full"`。
- 多筛选项页面使用 `<Row class="crud-search-grid">` 作为搜索区布局，普通输入、下拉、日期、日期范围统一走 `--crud-search-item-width` 宽度机制；不要再通过 `:xl="3/4/5/8"` 手工拉出不同长度。
- 下拉提示遵循同一宽度机制：最小宽度等于输入框宽度，按选项内容自动加宽，最大不超过 2 倍输入框宽度，超过后保持省略显示。
- 查询、重置等按钮放在最后一个 `<Col class="crud-search-grid__actions">`，按钮不使用 `block`，不占满一整列；窄屏也保持按钮自身宽度，避免搜索区按钮被拉宽后和输入项失去对齐关系。
- 新增、编辑、详情、完成、提测等业务表单不使用 `SearchField`，仍使用 `FormItem label + 控件`。
- 深色/浅色主题由 `crud-search-field.vue` 统一适配，业务页面不要额外写重复边框、前缀或高度样式。
- 该标准适用于所有存量和新增的列表页、报表页、日志页、配置列表；评审时检查是否存在 `SearchField` 未放入 `crud-search-grid`、标签不是 4 个汉字、按钮列未使用 `crud-search-grid__actions` 或按钮使用 `block` 的情况。

示例：

```vue
<script setup lang="ts">
import SearchField from '#/components/crud-search-field.vue';
</script>

<template>
  <Row class="crud-search-grid" :gutter="[16, 16]">
    <Col :xs="24" :sm="12" :xl="6">
      <SearchField label="搜索内容">
        <Input v-model:value="searchForm.keyword" allow-clear placeholder="请输入关键字" />
      </SearchField>
    </Col>
    <Col :xs="24" :sm="12" :xl="4">
      <SearchField label="当前状态">
        <Select v-model:value="searchForm.status" allow-clear class="w-full" placeholder="请选择">
          <SelectOption :value="1">启用</SelectOption>
          <SelectOption :value="0">禁用</SelectOption>
        </Select>
      </SearchField>
    </Col>
    <Col class="crud-search-grid__actions" :xs="24" :sm="12" :xl="6">
      <Space wrap>
        <Button type="primary" @click="handleSearch">搜索</Button>
        <Button @click="handleReset">重置</Button>
      </Space>
    </Col>
  </Row>
</template>
```

## 表格操作列标准

所有固定在右侧的表格操作列必须使用统一组件 `#/components/crud-table-actions.vue`，并配合 `estimateVisibleActionColumnWidth()` 估算稳定列宽。这里的“自动宽度”不是 Ant Design Vue 的 CSS `auto`，而是按当前可见操作按钮估算一个固定 `width`，保证 `fixed: 'right'` 横向滚动不抖动。

使用规则：

- 操作列继续保留 `fixed: 'right'`，但禁止随意写 `300px`、`360px` 等大宽度。
- 操作数组按业务优先级传入：常用操作在前，删除、彻底删除、取消等危险操作放最后。
- 可见操作数 `<= 3` 时平行展示；超过 3 个时组件自动展示前 2 个操作 + `更多` 下拉，`更多` 自身计入最多 3 个平行操作。
- `visible=false` 的操作不参与展示和宽度估算；行状态差异较大时用当前页数据 `items.map(rowActions)` 估算宽度。
- 删除、彻底删除、取消、发布、同步等需要确认的操作，优先在 action 上配置 `confirmTitle` / `confirmContent`；如果原处理函数内部已有 `Modal.confirm`，不要再重复配置确认。

示例：

```vue
<script setup lang="ts">
import { computed } from 'vue';
import CrudTableActions from '#/components/crud-table-actions.vue';
import { buildTableScrollX, estimateVisibleActionColumnWidth } from '#/utils/table';

const actionColumnWidth = computed(() =>
  estimateVisibleActionColumnWidth(items.value.length > 0 ? items.value.map(rowActions) : [rowActions({})]),
);
const columns = computed(() => [
  { title: '名称', dataIndex: 'name', width: 220 },
  { title: '操作', key: 'action', width: actionColumnWidth.value, fixed: 'right' as const },
]);
const tableScroll = computed(() => buildTableScrollX(columns.value, { minWidth: 960 }));

function rowActions(record: any) {
  return [
    { label: '查看', onClick: () => openView(record) },
    { label: '编辑', visible: canUpdate.value, onClick: () => openEdit(record) },
    { label: '删除', visible: canDelete.value, danger: true, confirmTitle: '确认删除该记录？', onClick: () => deleteRow(record.id) },
  ];
}
</script>

<template>
  <Table :columns="columns" :scroll="tableScroll">
    <template #bodyCell="{ column, record }">
      <template v-if="column.key === 'action'">
        <CrudTableActions :actions="rowActions(record)" />
      </template>
    </template>
  </Table>
</template>
```

## 按钮点击动作标准

所有会触发网络请求、文件导出、保存、删除、同步、批量处理的按钮，都必须提供明确的执行中反馈，并防止重复点击。纯前端交互（打开弹窗、切换页签、展开收起、复制本地文本）可以保留普通按钮，但不能影响异步按钮的状态管理。

使用规则：

- 表格行操作统一走 `#/components/crud-table-actions.vue`；`onClick` 可以返回 Promise，组件会等待异步完成，并在当前行操作执行中禁用同一行的其它操作。
- 表格行删除、取消、彻底删除、同步等危险或高影响操作，优先在 action 上配置 `confirmTitle` / `confirmContent`，确认框 OK 会等待异步 Promise 完成；如果旧函数内部已有复杂确认表单（例如重置密码输入框），不要重复套确认。
- 搜索、刷新、导出按钮必须绑定对应 `loading` / `exporting` 状态；重置按钮在列表加载中应禁用，避免搜索参数和列表数据出现错位。
- 保存、提交、同步、批量删除、批量恢复等写操作必须使用 `loading`、`confirm-loading` 或 `useAsyncAction()` 之类 pending 锁；失败后由现有请求拦截器或页面提示展示错误，状态必须在 `finally` 中释放。
- 登录按钮在 `loading` 中必须禁用，避免重复提交账号密码；业务插件独立登录页同样适用。
- 全局点击反馈样式由 `web/apps/web-antd/src/styles/crud-interactions.css` 维护，业务页面不要额外写会改变按钮尺寸或表格行高的点击动效。

示例：

```vue
<script setup lang="ts">
import { ref } from 'vue';

const loading = ref(false);
const exporting = ref(false);

async function load() {
  if (loading.value) return;
  loading.value = true;
  try {
    // 请求列表数据
  } finally {
    loading.value = false;
  }
}

async function handleExport() {
  if (exporting.value) return;
  exporting.value = true;
  try {
    // 调用 exportCrudXlsx()
  } finally {
    exporting.value = false;
  }
}
</script>

<template>
  <Button type="primary" :loading="loading" @click="load">搜索</Button>
  <Button :disabled="loading" @click="handleReset">重置</Button>
  <Button :loading="exporting" @click="handleExport">导出</Button>
</template>
```

## 权限控制

前端权限控制必须和后端一致：

| 场景 | 权限来源 |
|------|----------|
| 菜单是否显示 | 当前用户菜单树 |
| 页面是否可访问 | 路由和菜单权限 |
| 按钮是否显示 | 后端 `#[Auth]` code 对应的按钮权限 |
| 接口是否可调用 | 后端 AuthAspect 校验 |

前端隐藏按钮只是体验优化，不是安全边界。即使按钮被隐藏，后端接口仍必须通过 `#[Auth]` 校验。

System 后台和业务插件入口使用不同登录入口与 Token 缓存。后台页面请求 `/system/**`，业务插件前台请求自己的路由前缀，例如 `/demo/**`；新增前端页面时不要复用旧入口命名，也不要让 System Token 请求插件前台接口。插件入口由插件自己的 `auth-entry.ts` 声明，Web 编译期负责扫描，运行时只做识别和通用鉴权处理。

## 页面约定

- 列表页要提供搜索、分页、状态、回收站、前端导出等入口，并复用后端标准列表能力。
- 表格设置合理最小宽度，避免字段被挤压。
- 表格右侧操作列统一使用 `CrudTableActions` + `estimateVisibleActionColumnWidth()`；平行按钮最多 3 个，超过后自动收纳到“更多”下拉。
- 新增权限码后同步后端菜单、节点和前端按钮控制。
- 状态切换、删除、彻底删除、清空日志等高风险操作必须有明确二次确认。
- 表单字段名要与后端 `$fillable` 和 Service 校验保持一致。
- 时间范围参数应使用统一工具格式化，不要在页面中临时拼接不稳定格式。
- 前端导出要复用当前筛选条件和 index 分页接口，避免导出结果和页面结果不一致。

## 路由和菜单

如果使用后端菜单生成动态路由，需要确认：

- 菜单 `route` 与前端路由路径一致。
- 菜单 `component` 能映射到真实 Vue 页面。
- 目录菜单配置 `redirect`，避免点击目录后停留空白页。
- 外链和 iframe 菜单只用于可信地址。
- 按钮节点只用于权限控制，不应配置为页面路由。

## 插件前端

推荐方式：

- API service、私有组件和页面放在插件自己显式配置的 `plugin.view_root` 下。
- `plugin.json` 声明 `apps/menus/permissions`，菜单写 `route` 和相对 `view`。
- 菜单 code、按钮 code 与后端 `#[Auth]` code 保持一致。
- 插件页面仍参与主应用 typecheck/build，不单独绕过前端构建；当前机制不是运行时远程插件加载。即使通过 SmartAdminLibrary 提供的 `xadmin:plugin:*` 源码命令安装或恢复插件，前端变化后仍要重新执行 `composer web:build`。
- 插件页面、`routes.ts` 或 `auth-entry.ts` 变化后必须重新执行 `composer web:build`；发布打包时 `composer build` 只复用并校验已有 `web/dist`。

插件包页面：

- 独立插件包可携带 `plugin/<Module>/<view_root>` 页面，以及清单声明的语言包、迁移等资料文件。
- Web 运行库会在编译期读取 `plugin/*/plugin.json`，只按已配置的 `plugin.view_root` 自动注册插件 Vue 页面；未配置时不参与打包映射。
- 后端菜单同步把 `view` 统一转换为 `@plugin/<Module>/views/<path>.vue`，数据库不需要保存真实目录。
- 插件页面内的 API 和组件建议使用相对路径导入；通用组件、布局和运行依赖复用 Web 公共包。
- 插件独立登录页通过 `plugin/<Module>/<view_root>/auth-entry.ts` 或 `auth-entry.mts` 声明，不能在 Web 壳中为具体插件写死路由。

示例：

```json
{
  "plugin": {
    "code": "demo",
    "name": "演示插件",
    "view_root": "stc/view",
    "language_root": "stc/languages",
    "migration_root": "stc/migrations"
  },
  "apps": [
    {
      "id": 9000,
      "name": "演示应用",
      "route": "/demo",
      "component": "BasicLayout",
      "menus": [
        {
          "id": 9010,
          "name": "订单管理",
          "code": "demo.order.index",
          "route": "/demo/order",
          "view": "order/index.vue"
        }
      ]
    }
  ]
}
```

### 插件认证入口配置

`auth-entry.ts` 用于描述插件前台入口的登录页、认证 API、路由范围、权限码前缀、用户模型和可选菜单。Web 的 Vite 虚拟模块会扫描所有配置了 `plugin.view_root` 的插件，把入口合并到统一前端入口配置。

示例：

```ts
export default {
  authBase: '/demo/account/auth',
  entry: 'demo',
  homePath: '/demo/order',
  loginPath: '/demo/login',
  menus: [],
  name: '演示应用',
  permissionPrefixes: ['demo.'],
  routePrefixes: ['/demo'],
  userModel: 'Plugin\\Demo\\Model\\DemoAccount',
  userModelIncludes: ['DemoAccount'],
};
```

前端约束：

- `loginPath`、`homePath` 和 `routePrefixes` 必须互相匹配，避免登录后 redirect 逃逸到其他入口。
- 每个入口单独缓存 Token，切换入口时只恢复当前入口 Token，避免 System 后台和插件前台互相污染。
- `coreAuthApiService.getUiMeta()` 使用公开的 `/system/auth/ui-meta`，适合登录页和启动初始化。
- `dataApiService.getUiMeta()` 对应 `/system/data/ui-meta`，只用于已登录后的界面元信息和系统设置场景；不要在插件登录页初始化时直接调用它。
- 插件菜单可由 `auth-entry.ts` 的 `menus` 提供，Web 会按当前入口权限码裁剪；后台管理菜单仍由 `plugin.json` 和 System RBAC 控制。

## 联调清单

开发完成后按以下顺序检查：

1. 后端接口能用登录 Token 调通。
2. 菜单/节点同步后，管理员能在角色授权树看到新页面和按钮。
3. 给普通角色授权后，普通用户重新登录能看到菜单。
4. 页面搜索、分页、新增、编辑、删除、恢复、前端导出和状态切换与接口一致。
5. 取消某个按钮权限后，按钮隐藏且直接请求接口返回 403。
6. 数据范围收紧后，列表、详情、更新、删除都不能越权。

## 常见问题

- 页面 404：菜单 route 未同步、角色未授权或动态路由未刷新。
- 页面空白：`plugin.view_root`、菜单 `view` 或真实 Vue 文件路径不匹配。
- 按钮显示但请求 403：前端按钮权限和后端权限码不一致。
- 插件登录后反复回到登录页：检查 `auth-entry.ts` 是否声明了当前路由范围，`profile` 返回的 `auth_user_model` 是否能匹配 `userModel` 或 `userModelIncludes`。
- 插件登录页提示 `system.data.ui-meta` 未登录：确认登录页初始化使用 `coreAuthApiService.getUiMeta()`，不要在登录前调用需要 Token 的系统数据接口。
- 请求成功但页面无数据：响应结构没有按 `code/info/data/path` 或分页结构解析。
- 构建后 API 地址错误：检查 `_app.config.js` 生成逻辑和生产域名配置。
- 插件页面开发环境可访问但构建失败：检查 `plugin.json` 是否能被 Web 读取、页面类型是否完整、插件是否使用了 Web 未提供的依赖。
