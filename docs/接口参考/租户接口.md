# 租户接口

控制器：`plugin/System/src/Controller/TenantController.php`
前缀：`/system/tenant`

| 方法 | 路径 | 认证 | 权限码 | 日志 | 用途 |
|------|------|------|--------|------|------|
| GET | `/system/tenant/index` | 权限 | `system.tenant.index` | 否 | 租户列表 |
| GET | `/system/tenant/recycle` | 权限 | `system.tenant.index` | 否 | 租户回收站 |
| GET | `/system/tenant/info/{id}` | 权限 | `system.tenant.index` | 否 | 租户详情 |
| POST | `/system/tenant/create` | 权限 | `system.tenant.create` | 新增租户 | 新增租户 |
| PUT | `/system/tenant/update/{id}` | 权限 | `system.tenant.update` | 编辑租户 | 编辑租户 |
| DELETE | `/system/tenant/delete/{ids}` | 权限 | `system.tenant.delete` | 删除租户 | 删除到回收站 |
| DELETE | `/system/tenant/real-delete/{ids}` | 权限 | `system.tenant.real-delete` | 彻底删除租户 | 彻底删除 |
| PUT | `/system/tenant/recovery/{ids}` | 权限 | `system.tenant.recovery` | 恢复租户 | 恢复 |
| PUT | `/system/tenant/status/{id}` | 权限 | `system.tenant.update` | 更新租户状态 | 启用/禁用 |
| GET | `/system/tenant/statistics` | 权限 | `system.tenant.index` | 否 | 租户统计 |
| GET | `/system/tenant/options` | 权限 | `system.tenant.index` | 否 | 租户选项 |

## 使用说明

租户管理属于平台能力。普通业务查询依赖登录后建立的 `TenantContext`，不要通过业务接口手工伪造租户上下文。

## 查询参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `page` / `pageSize` | number | 分页参数 |
| `code` | string | 租户编码 |
| `name` | string | 租户名称 |
| `contact_name` | string | 联系人 |
| `contact_phone` | string | 联系电话 |
| `status` | number | 租户状态 |
| `expired_at` | string[] | 到期时间范围 |

列表、回收站、详情和统计都属于平台租户管理能力，应只授予平台维护人员；前端导出复用租户列表接口。

## 写入字段

| 字段 | 创建 | 编辑 | 说明 |
|------|------|------|------|
| `code` | 必填 | 通常不改 | 租户唯一编码，建议使用稳定英文/数字编码 |
| `name` | 必填 | 可改 | 租户名称 |
| `contact_name` | 可选 | 可改 | 联系人 |
| `contact_phone` | 可选 | 可改 | 联系电话 |
| `contact_email` | 可选 | 可改 | 联系邮箱 |
| `package_code` | 可选 | 可改 | 套餐编码或版本标识 |
| `expired_at` | 可选 | 可改 | 到期时间 |
| `status` | 可选 | 可改 | 启用状态 |
| `remark` | 可选 | 可改 | 维护备注 |

## 状态和删除

- `status/{id}` 用于启用或禁用租户。禁用租户后，登录和业务访问应按服务端租户上下文规则处理。
- `delete/{ids}` 是软删除，进入回收站。
- `recovery/{ids}` 从回收站恢复。
- `real-delete/{ids}` 是彻底删除，高风险操作，执行前要确认租户数据、上传文件、审计日志和备份策略。

## 租户上下文

登录成功后，系统会建立租户上下文。带 `tenant_id` 且在 `$fillable` 中声明的模型会自动接入租户范围。二次开发时需要注意：

- 支持租户隔离的业务表必须有 `tenant_id` 字段。
- Model 的 `$fillable` 必须包含 `tenant_id`，否则不会按通用租户范围处理。
- 普通业务查询不要手工拼接租户条件，优先复用 `CoreModel`、`CoreMapper` 和 `TenantContext`。
- 超级管理员逻辑要显式判断，不要用 `tenant_id = 0` 作为隐式平台数据兜底。
