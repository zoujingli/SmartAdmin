# 在线测试

接口参考页统一采用“接口表格 + 接口案例 + 在线调试”的结构：表格负责快速索引，案例负责解释参数，案例下方只显示一个“在线调试”按钮。点击按钮后会弹出统一调试框，并自动带入当前接口的 `method`、`path`、`headers`、`query` 和 `body`。

当前文档已经为每个接口生成以下内容：

1. `<!-- API_CASE: METHOD /path -->` 标记，用于静态检查确认接口案例覆盖率。
2. `jsonc` 请求案例，说明 `method`、`path`、`headers`、`query`、`body` 以及业务字段含义。
3. `api-test` 在线调试配置，docsify 会把它渲染成一个调试按钮，点击后打开弹窗。

## 使用场景

- 快速验证本地后端接口。
- 给开源用户提供可交互的接口示例。
- 在接口文档中展示请求体、查询参数、认证方式和业务边界。
- 排查 Token、CORS、路径、请求体格式等联调问题。
- 连续调试登录、列表、详情、写入、删除等接口，并复用同一个 Base URL、Token 和测试账号；前端导出不再对应独立后端接口。

## 调试流程

1. 打开任意接口参考页面，找到接口案例下方的“在线调试”按钮。
2. 点击按钮后，在弹窗中确认 Base URL、Path、Method、登录入口、Headers、Query 和 Body；本地预览默认使用 `http://127.0.0.1:9501`，避免只显示占位符但请求时缺少 Base URL。
3. 登录接口会默认带入 `admin/admin`，Project 登录案例会优先使用案例中的 `project_user`；也可以在弹窗的 Username、Password 中修改。
4. 任意非登录接口发送前，如果没有 Token 且存在缓存账号密码，工具会按路径识别登录入口：`/project/**` 使用 Project 应用入口，其它接口使用 System 后台入口；随后先获取对应 `password-crypto` 参数，再用加密后的密码自动登录获取 Token。
5. 如果接口返回 HTTP `401` 或业务 `code=401`，工具会清理旧 Token，并在有账号密码时自动登录后重试当前接口一次。
6. 登录、刷新接口识别到新 Token 后会自动更新缓存；退出登录成功后只清理 Token，保留 Base URL 和账号密码。

> 密码说明：后台用户和 Project 账号密码在数据库中都是哈希值；接口传输时必须是 `{kid, nonce, ciphertext}` 加密对象。在线调试弹窗默认使用 `server-rsa`：发送前按登录入口调用 `/system/auth/password-crypto` 或 `/project/account/auth/password-crypto` 获取公钥和 nonce，再用 RSA-OAEP(SHA-1) 加密 Password 输入框或案例中的明文测试值。
> 密钥说明：服务端 RSA 私钥由系统自动维护在运行目录，公钥由接口实时下发，部署时不需要单独配置证书、公钥或私钥路径。若运行环境缺少系统 `openssl.cnf`，服务会在密钥目录自动生成内部最小配置文件，在线调试无需额外设置。

## 安全边界

- 请求从当前浏览器发出，受浏览器 CORS 策略限制。
- Base URL、Token、Username、Password 会使用浏览器 Web Crypto 的 AES-GCM 加密后保存到 `localStorage`。
- 加密缓存只避免 Token 和密码以明文形式落入 `localStorage`；它不能抵御同源 XSS、恶意浏览器插件或已被控制的用户浏览器。
- 本地加密缓存和服务端密码哈希是两件事：前者只保护文档调试工具的浏览器缓存，后者保护数据库中的用户密码。
- 默认开启“加密记住账号密码”，便于开源用户连续调试；公开环境或生产账号不建议使用该功能。
- 点击“清除记忆”会清空加密缓存和密钥，并恢复默认账号密码 `admin/admin`。
- 不要在公开文档中写入真实 Token、密码、密钥、生产个人账号或生产业务数据。
- 删除、彻底删除、恢复、授权、改密、发布升级等接口发送前要确认环境，建议只在本地或测试环境验证。

## 案例写法

接口案例先写给人看的 `jsonc`，每个字段都要写明用途、是否必填、取值边界、权限或数据范围影响。

````markdown
```jsonc
{
  // method: HTTP 方法。必须与接口表格中的方法一致。
  "method": "POST",
  // path: 请求路径。路径参数需要替换为真实 ID。
  "path": "/system/auth/login",
  // headers: 请求头。登录接口只需要 JSON；登录态接口会自动追加 Authorization。
  "headers": {
    "Content-Type": "application/json"
  },
  // query: Query 参数。无查询条件时传空对象。
  "query": {},
  // body: Body 请求体。字段注释需要说明业务语义和安全边界。
  "body": {
    // username: 必填，登录用户名。
    "username": "admin",
    // password: 必填，RSA-OAEP 加密对象；这是最终发送到后端的结构。
    "password": {
      // kid: password-crypto 返回的密钥标识。
      "kid": "runtime-key-id",
      // nonce: password-crypto 返回的一次性随机数，服务端会立即消费。
      "nonce": "server-issued-nonce",
      // ciphertext: 加密 {"v":1,"purpose":"system.auth.login.password","nonce":"...","ts":当前秒级时间戳,"value":"真实密码"} 后的 base64。
      "ciphertext": "base64-rsa-oaep-ciphertext"
    }
  }
}
```
````

案例下方再放同结构的 `api-test`。调试器支持 JSONC 注释，会在渲染前移除 `//` 与 `/* ... */` 注释并解析为 JSON。

````markdown
```api-test
{
  // name: 调试弹窗标题。
  "name": "登录接口",
  // method: HTTP 方法。
  "method": "POST",
  // path: 调试路径。
  "path": "/system/auth/login",
  // headers: 请求头。
  "headers": {
    "Content-Type": "application/json"
  },
  // query: Query 参数。
  "query": {},
  // body: Body 请求体。
  "body": {
    // username: 登录用户名。弹窗发送登录请求时会以 Username 输入框为准。
    "username": "admin",
    // password: 调试输入明文示例；弹窗发送时会自动替换为 RSA-OAEP 加密对象。
    "password": "admin"
  }
}
```
````

## 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | string | 测试块标题，建议写成“用途 · 路径”，便于长页面区分接口 |
| `method` | string | HTTP 方法，支持 GET、POST、PUT、PATCH、DELETE、HEAD |
| `path` | string | 接口路径，可以是 `/system/auth/login`，也可以是完整 URL |
| `baseUrl` | string | 可选，默认 Base URL；不填时优先使用本地加密缓存值，本地预览默认回退到 `http://127.0.0.1:9501` |
| `authEntry` | string | 可选，`auto/system/project`；默认 `auto`，按路径自动识别 System 后台或 Project 应用登录入口 |
| `headers` | object | 请求头对象；未手写 `Authorization` 或值为 `Bearer <token>` 时会自动替换为当前 Token |
| `query` | object | 查询参数对象；发送时会拼接到 URL 查询串 |
| `body` | object/string | 请求体；对象按 JSON 发送，字符串按原始文本发送 |
| `passwordTransport` | string | 可选，默认 `server-rsa`；`body` 表示完全使用 Body 中已有的加密对象，不再自动生成 |

## 密码加密调试

调试器会识别以下密码入口并自动按用途生成密文：

| 接口 | 字段 | purpose |
|------|------|---------|
| `POST /system/auth/login` | `password` | `system.auth.login.password` |
| `PUT /system/auth/password` | `old_password` | `system.auth.password.old_password` |
| `PUT /system/auth/password` | `new_password` | `system.auth.password.new_password` |
| `POST /project/account/auth/login` | `password` | `system.auth.login.password` |
| `PUT /project/account/auth/password` | `old_password` | `system.auth.password.old_password` |
| `PUT /project/account/auth/password` | `new_password` | `system.auth.password.new_password` |
| `POST /system/user/create` | `password` | `system.user.create.password` |
| `PUT /system/user/update/{id}` | `password` | `system.user.update.password` |
| `PUT /system/user/reset-password/{id}` | `password` | `system.user.reset_password.password` |

`jsonc` 请求案例展示最终请求对象，便于用户理解真实协议；`api-test` 配置可以保留明文测试值，调试器会在请求发出前替换为 `{kid, nonce, ciphertext}`。响应区域只展示真实 URL、HTTP 状态、业务 `code` 和响应 JSON，不会输出本地输入的明文密码。

## 本地加密缓存

在线测试会使用以下 `localStorage` 键值：

| Key | 说明 |
|-----|------|
| `smartadmin.docs.secure.v1` | AES-GCM 加密后的 Base URL、Token、用户名、密码、密码传输方式和记忆开关 |
| `smartadmin.docs.secure.key.v1` | 当前文档域名下生成的 AES 原始密钥，用于后续解密缓存 |

登录接口返回项目标准结构时，调试器会自动识别以下字段：

| 字段 | 说明 |
|------|------|
| `data.token` | SmartAdmin 当前登录接口返回的访问令牌 |
| `data` 字符串 | 当前刷新接口返回的访问令牌 |

识别到新 Token 后会同步到加密缓存和当前弹窗，后续接口会自动携带 `Authorization: Bearer <token>`。

## CORS 配置

在线测试从文档站域名向 API 域名发起请求。如果文档站和 API 不同源，后端必须允许文档站域名跨域访问。例如本地文档站是 `http://127.0.0.1:18100`，后端 API 是 `http://127.0.0.1:9501`，后端 CORS 需要允许该 Origin。

常见错误：

| 现象 | 原因 |
|------|------|
| 请求失败且无响应体 | CORS 未放行或 HTTPS 证书不被浏览器信任 |
| 获取密码加密参数失败 | 后端 `/system/auth/password-crypto` 或 `/project/account/auth/password-crypto` 不可访问、runtime 密钥目录不可写，或服务端 OpenSSL 扩展异常 |
| 401 | Token 缺失、过期、无效或自动登录失败 |
| 403 | 当前用户没有接口权限 |
| 500 | 业务参数不符合后端校验 |
| Mixed Content | HTTPS 文档页请求 HTTP API，被浏览器拦截 |

## 维护要求

- 每个接口表格行都必须有一个对应的 `<!-- API_CASE: METHOD /path -->`。
- 每个接口案例都必须包含 `jsonc` 请求案例和 `api-test` 在线调试配置。
- `jsonc` 中 `method`、`path`、`headers`、`query`、`body` 必须有字段注释，业务字段也应尽量逐项说明。
- 修改接口路径、请求体、认证方式时同步更新接口案例。
- `composer docs:check` 会检查接口案例、调试配置和基础 docsify 配置，提交前必须通过。
