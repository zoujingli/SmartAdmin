# 接口参考

接口路径按 Hyperf Controller prefix 和 Mapping 注解整理。默认响应体使用项目统一结构：`code/info/data/path`。

## 基础约定

| 项目 | 约定 |
|------|------|
| Base URL | 文档在线调试默认使用 `https://smart.thinkadmin.top`；本地联调时手动改为 `http://127.0.0.1:9501` 或实际后端地址 |
| Content-Type | JSON 接口使用 `application/json`，上传接口使用 `multipart/form-data` |
| 认证方式 | 登录后携带访问令牌，通常使用 `Authorization: Bearer <token>` |
| 时间格式 | 日期使用 `YYYY-MM-DD`，日期时间使用 `YYYY-MM-DD HH:mm:ss`；新增跨字段日期/时间范围统一使用 `start_date`、`after_date` |
| 分页参数 | 通常使用 `page`、`pageSize` |
| 响应字段 | 固定包含 `code`、`info`、`data`、`path` |

## 接口真源与边界

接口参考是前后端和第三方联调的合同源头，但合同必须来自当前实现：

| 来源 | 复核重点 |
|------|----------|
| Controller | prefix、Mapping、`#[Auth]`、`#[Logger]`、path 参数和公开入口 |
| Core 继承 | `CoreController`、`CoreService`、`CoreMapper` 提供的标准 CRUD、状态、软删、恢复、彻底删除和 options 能力 |
| Service | `_vali()` 校验、唯一性、事务、状态流转、签名或外部调用规则 |
| Mapper | 查询白名单、分页、排序、数据范围、租户范围和列表后处理 |
| Model | `$fillable`、`$hidden`、类型转换、软删、关联和审计字段 |
| 前端页面 | 表单必填、按钮权限、二次确认、导出和上传行为 |

后台 `/system/**` 接口继续使用登录 Token + RBAC。插件前台接口和开放接口按各自文档声明的认证方式处理，不能混用 System Token、插件 Token 和开放接口 Token。


## 接口页标准

接口参考采用“目录页 + 字段级说明 + 可调试示例”的维护方式，具体页面尽量自包含，避免只写路径列表。新增或调整接口时，请同时遵守 [接口文档规范](../文档维护/接口文档规范.md)。

| 区块 | 要求 |
|------|------|
| 接口内容 | 写清接口组边界、业务用途和不覆盖的场景 |
| 基本信息 | 写明方法与路径、Controller 方法、权限码、鉴权方式和操作日志 |
| 请求字段 | 按 `path/query/body/form/files/headers` 拆表，标明类型和必填性 |
| 请求示例 | 使用 `jsonc`，关键字段带中文注释，去掉注释后可解析 |
| 响应示例 | 使用 `jsonc`，保留 `code/info/data/path` 标准响应壳，并展开本接口 `data` 的可对接字段 |
| 关键约束 | 写明租户、数据范围、敏感字段、幂等、金额单位、状态流转等 |
| 源码来源 | 列出 Controller、Service、Mapper、Model、前端页面或插件清单 |

每个具体接口案例固定按“请求 `jsonc` → `api-test` 在线调试配置 → 响应 `jsonc`”维护。请求示例需要精确列出 `headers/query/body` 字段并写中文注释；响应示例需要写清 `code/info/data/path`，列表接口展示 `items/pageInfo`，选项/树接口展示数组结构，详情和写操作展示对象或已处理 ID，不能用“见字段说明”替代具体字段。

## 认证说明

| 标记 | 含义 |
|------|------|
| 公开 | 不要求登录 |
| 登录 | 需要有效 Token |
| 权限 | 需要有效 Token 且通过 `#[Auth]` 权限校验 |

写操作通常带 `#[Logger]`，会记录操作日志；敏感字段按注解和全局脱敏规则过滤。

System 后台和插件前台使用独立登录入口与 Token 缓存。`/system/auth/ui-meta` 是公开的低敏界面元信息接口；`/system/data/ui-meta` 是登录态元信息接口，接受实现 `UserModelInterface` 的有效用户模型，但不能代表其他 `/system/**` 业务接口对插件 Token 开放。

## 插件分发标识

公开文档站保留全生态插件文档，便于用户评估能力、联调接口和规划授权交付；但文档可见不代表插件源码或 ZIP 已随开源版分发。标记为“会员授权”的页面只展示接口与使用说明，安装使用需会员授权或内部交付。

| 标识 | 文档可见 | 源码/ZIP 分发 | 安装与调试 |
|------|----------|---------------|------------|
| 开源 | 是 | 随公开 SmartAdmin 仓或公开插件包分发 | 可直接按开源协议安装、运行和联调 |
| 会员授权 | 是 | 不随开源版源码或公开 Release ZIP 分发 | 需会员授权或内部交付后，在源码环境安装并构建 |
| 私有交付 | 按项目约定 | 不进入公开分发链路 | 由客户项目或私有 ZIP 维护 |

## 开放接口认证口径

- 普通后台接口：使用 `Authorization: Bearer <token>`，权限由 `#[Auth]` 与菜单按钮 code 控制。
- 插件独立前台接口：使用插件自己的登录入口和 Token 缓存，不能直接复用 System 后台 Token。
- AI 辅助接口只服务后台运营录入，返回候选内容和差异，不直接写库、不自动发布，也不进入 `/website/**` 开放接口。

## 统一响应

成功响应示例：

```jsonc
{
  // code: 业务响应码；成功固定为 200。
  "code": 200,
  // info: 业务提示文案，字段名固定为 info。
  "info": "获取成功",
  // data: 具体接口返回数据；详情页为对象，列表页为 items/pageInfo。
  "data": {
    // id: 示例记录 ID。
    "id": 1
  },
  // path: 当前请求路径。
  "path": "/system/user/index"
}
```

失败响应示例：

```jsonc
{
  // code: 业务响应码；无权限固定为 403。
  "code": 403,
  // info: 失败提示文案。
  "info": "无操作权限",
  // data: 失败时通常为空，部分业务会返回辅助定位数据。
  "data": null,
  // path: 当前请求路径。
  "path": "/system/user/delete/1"
}
```

HTTP status 固定返回 `200`；业务状态统一读取 `body.code`。项目标准业务码只使用：

| code | 含义 | 使用边界 |
|------|------|----------|
| `200` | 成功 | 接口处理成功 |
| `401` | 未认证 | Token 缺失、过期、无效或刷新失败 |
| `403` | 无权限 | Token 有效但没有接口权限 |
| `404` | 路由不存在 | 页面或 API 路由不存在 |
| `500` | 业务失败或服务异常 | 账号密码错误、业务校验失败、状态不允许、数据不存在等 |

不要把账号密码错误返回 `401`，也不要把业务数据不存在返回 `404`。这些边界会影响前端请求拦截器、登录态处理和错误提示。

## 分页结构

分页接口的 `data` 通常包含列表、总数和分页信息。不同模块可能有额外统计字段，页面应通过 API service 统一解析，避免在业务组件中硬编码响应结构。

常见结构：

```jsonc
{
  // code: 业务响应码；成功固定为 200。
  "code": 200,
  // info: 业务提示文案。
  "info": "获取成功",
  // data: 分页数据主体。
  "data": {
    // items: 当前页数据列表。
    "items": [
      {
        // id: 记录 ID。
        "id": 1,
        // name: 记录名称。
        "name": "示例名称"
      }
    ],
    // pageInfo: 标准分页信息，由 CoreMapper 返回。
    "pageInfo": {
      // total: 总记录数。
      "total": 0,
      // totalPage: 总页数。
      "totalPage": 0,
      // currentPage: 当前页码。
      "currentPage": 1
    },
    // extra: 可选扩展统计，不是所有列表都会返回。
    "extra": {}
  },
  // path: 当前请求路径。
  "path": "/system/user/index"
}
```

## 列表查询规范

- 查询字段必须由 Mapper 白名单处理。
- 新增日期/时间范围参数统一使用 `start_date`、`after_date` 表示闭区间边界；已有字段级 `QueryHelper::dateBetween()` 可继续使用具体字段名承载范围字符串或数组。
- 请求参数不能直接控制 raw SQL、select、orderBy、关联名或表达式。
- 前端导出应复用 index 列表筛选条件；数据范围由标准列表接口保证。
- 统计、选项、详情、更新、删除等非列表接口也要考虑数据范围。

## 写操作规范

写接口一般满足：

- 需要 `Auth::CHECK`。
- 有明确权限码。
- 添加 `#[Logger]`。
- 敏感字段通过 `excludeFields` 或全局脱敏规则处理。
- Service 层完成唯一性校验和事务。
- Mapper/Model 不绕过 fillable、租户和数据范围。

高风险写操作包括彻底删除、清空日志、发布升级、上传配置、角色授权、用户密码重置。调用这些接口时前端应提供明确确认。

## 权限码和日志

| 字段 | 说明 |
|------|------|
| 权限码 | 文档表格中的 `权限码` 列，必须与 `#[Auth]` 和菜单按钮 code 对齐 |
| 认证 | `公开`、`登录`、`权限` 三种 |
| 日志 | `#[Logger]` 名称或是否记录日志 |
| 用途 | 接口业务用途，不代表完整校验规则 |

如果表格写“默认创建权限”或“默认编辑权限”，表示该接口使用项目当前注解默认 code 生成规则或 Controller 约定；二次开发时建议显式写 code，降低同步歧义。

## 调试顺序

接口调试建议按以下顺序：

1. 登录获取 Token。
2. 调用 `/system/auth/profile` 确认用户信息。
3. 调用 `/system/auth/codes` 或菜单权限接口确认权限码。
4. 调用目标接口。
5. 若返回 403，检查角色授权和节点同步。
6. 若列表为空，检查数据范围、租户上下文和查询条件。
7. 若写操作失败，查看操作日志和后端异常日志。

如果需要在文档页直接发请求，可以使用 [在线测试](./在线测试.md)。在线测试从浏览器发起请求，受 CORS、HTTPS 证书和 Token 权限限制。

## 文档维护规则

新增或修改接口时同步更新本章节：

- 路径、HTTP 方法和 Controller 前缀。
- 认证方式、权限码、日志名称。
- 查询参数、写入字段和请求体示例。
- 数据范围、租户隔离、敏感字段和审计说明。
- 前端 API service 文件路径。
- 插件页面所属 `plugin.json` 和显式资源根字段。
- `composer docs:check` 会直接比对后端 Controller 注解与接口参考表格，确保接口数量和路径不漂移。
