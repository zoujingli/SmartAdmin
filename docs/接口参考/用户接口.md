## 使用说明

- 用户列表、详情、关系读取和写入都受数据范围保护。
- 角色、部门、岗位关系分配会记录关系变更日志。
- 密码字段必须脱敏，不允许返回或记录明文。
- 新增、编辑、重置密码接口接收的是 RSA-OAEP 加密后的密码对象；服务端解密后写入 `SystemUser.password` 时会通过模型访问器执行 `password_hash()`。明文字符串会被拒绝，避免日志、代理或浏览器调试中出现真实密码。

## 列表查询

`index` 和 `recycle` 使用分页查询，常见参数如下。实际查询条件以后端 Mapper 白名单为准，前端不应传入任意 SQL 字段、排序表达式或关联名。

| 参数 | 类型 | 说明 |
|------|------|------|
| `page` | number | 页码，默认由前端分页组件传入 |
| `pageSize` | number | 每页数量 |
| `username` | string | 用户名模糊查询 |
| `nickname` | string | 昵称模糊查询 |
| `phone` | string | 手机号查询 |
| `email` | string | 邮箱查询 |
| `status` | number | 启用状态 |
| `dept_id` | number | 部门过滤，仍受当前登录人数据范围限制 |

返回数据通常包含用户基础字段、部门/角色/岗位展示字段、创建更新时间和分页信息。密码永远不返回。

## 写入字段

创建和编辑接口接收用户基础资料。新增用户需要提供登录必要字段，编辑时不建议通过普通编辑接口重置密码，重置密码应走专用接口。

| 字段 | 创建 | 编辑 | 说明 |
|------|------|------|------|
| `username` | 必填 | 通常不改 | 登录用户名，必须唯一 |
| `password` | 必填 | 可选 | RSA-OAEP 加密对象，包含 `kid`、`nonce`、`ciphertext`，服务端解密后自动哈希，日志排除 |
| `nickname` | 必填 | 可改 | 展示昵称 |
| `phone` | 可选 | 可改 | 联系电话 |
| `email` | 可选 | 可改 | 邮箱 |
| `avatar` | 可选 | 可改 | 头像 URL |
| `signed` | 可选 | 可改 | 个性签名 |
| `status` | 可选 | 可改 | 启用状态 |
| `remark` | 可选 | 可改 | 管理备注 |

## 关系接口

关系接口用于独立维护用户和角色、部门、岗位的多对多关系。保存关系时建议提交 ID 数组，后端会以当前提交结果为准同步关系。

| 接口 | 请求体示例 | 返回含义 |
|------|------------|----------|
| `PUT /system/user/roles/{id}` | `{ "role_ids": [1, 2] }` | 保存用户角色 |
| `PUT /system/user/depts/{id}` | `{ "dept_ids": [3, 4] }` | 保存用户部门 |
| `PUT /system/user/posts/{id}` | `{ "post_ids": [5] }` | 保存用户岗位 |
| `GET /system/user/roles/{id}` | 查询参数为空 | 返回已选角色 ID 或角色列表 |
| `GET /system/user/depts/{id}` | 查询参数为空 | 返回已选部门 ID 或部门列表 |
| `GET /system/user/posts/{id}` | 查询参数为空 | 返回已选岗位 ID 或岗位列表 |

> 字段名以服务端当前实现为准；前端封装位于 `web/apps/web-antd/src/api/system/user.ts`，二次开发时应优先复用该封装。

## 状态、排序和密码

| 操作 | 建议请求体 | 说明 |
|------|------------|------|
| 更新状态 | `{ "status": 1 }` | 只修改启用状态，权限码为 `system.user.update` |
| 更新排序 | `{ "sort": 10 }` | 用于列表排序 |
| 重置密码 | `{ "password": { "kid": "runtime-key-id", "nonce": "...", "ciphertext": "..." } }` | 提交 RSA-OAEP 加密对象，日志排除密码字段，服务端解密后负责哈希 |

## 权限与审计

- 创建、编辑、删除、恢复、彻底删除、状态、排序、密码重置、关系分配都会记录操作日志。
- `create`、`update`、`reset-password` 明确排除 `password` 字段。
- 普通用户只能操作数据范围内的用户；超级管理员或拥有全部数据范围的角色也应通过标准接口访问，避免绕过租户和数据范围。

### 4. 新增用户

<!-- API_CASE: POST /system/user/create -->

`POST /system/user/create`

- 认证要求：权限
- 权限码：默认用户创建权限
- 操作日志：新增系统用户，排除 password
- 用途说明：新增用户

请求案例：

```jsonc
{

  // path: 请求路径。文档中的 {id}、{ids} 为占位符，调试时需要替换为真实 ID 或逗号分隔 ID 列表。
  "path": "/system/user/create",
  // headers: 请求头。登录态接口需要 Authorization: Bearer <token>，在线调试器会自动带入已保存 Token。
  "headers": {
    "Content-Type": "application/json",
    "Authorization": "Bearer <token>"
  },
  // query: Query 参数。GET 列表、统计、选项接口主要使用该对象；没有查询条件时传空对象。
  "query": {
  },
  // body: Body 请求体。POST/PUT/PATCH 写操作主要使用该对象；GET/DELETE 通常为空字符串。
  "body": {
    // username: 创建必填，编辑通常不建议修改。登录用户名必须唯一。
    "username": "demo_user",
    // password: 创建必填，RSA-OAEP 加密对象；purpose 固定为 system.user.create.password。
    "password": {
      // kid: password-crypto 返回的密钥标识。
      "kid": "runtime-key-id",
      // nonce: 创建用户密码独占的一次性随机数。
      "nonce": "nonce-for-user-create-password",
      // ciphertext: 加密创建用户密码载荷后的 base64 密文。
      "ciphertext": "base64-rsa-oaep-ciphertext"
    },
    // nickname: 必填，用户显示名称。
    "nickname": "演示用户",
    // email: 可选，用户邮箱。
    "email": "demo@example.com",
    // phone: 可选，用户手机号。
    "phone": "13800000000",
    // deptId: 可选，主部门 ID。前端表单字段，后端服务会同步到部门关系。
    "deptId": 1,
    // roleIds: 可选，角色 ID 数组。用于初始化或更新用户角色。
    "roleIds": [1],
    // postIds: 可选，岗位 ID 数组。用于初始化或更新用户岗位。
    "postIds": [1],
    // status: 可选，账号状态。1 启用，0 禁用。
    "status": 1,
    // remark: 可选，管理员备注。
    "remark": "测试账号"
  }
```
