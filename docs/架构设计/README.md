# 架构设计

本章解释 SmartAdmin 为什么按插件化、分层、权限节点、租户上下文和发布快照来组织。它不是代码清单，而是二次开发前必须理解的系统边界。

## 架构目标

SmartAdmin 的架构目标是让新模块接入时自动获得通用后台能力，同时避免核心模块长期失控：

- 插件自治：模块通过 `composer.json`、`plugin.json` 和 Provider 分工声明安装、应用清单与运行期能力，发布二进制只运行已打包插件。
- 分层清晰：Controller、Service、Mapper、Model 各自承担固定职责。
- 权限可治理：菜单、按钮、接口注解统一收敛为权限节点。
- 数据可隔离：数据范围和租户上下文在后端查询层生效。
- 发布可回滚：数据库快照、dry-run 和备份恢复形成发布闭环。

## 与开发指南的关系

架构设计回答“为什么这样组织”，开发指南回答“新增模块时怎么写”。如果你准备新增业务模块，建议先读本章，再读 [新模块开发](../开发指南/新模块开发.md)。

## 与系统功能的关系

[系统功能](../系统功能/README.md) 解释“系统现在提供什么能力”，本章解释“这些能力为什么按这种方式实现”。例如：

| 功能问题 | 架构问题 |
|----------|----------|
| 用户如何分配角色 | 角色、节点、菜单和用户关系如何建模 |
| 文件如何上传 | 上传链路如何和驱动解耦 |
| 日志如何记录 | 注解、切面、模型变更如何协作 |
| 租户如何隔离 | 租户上下文如何进入 Model/Mapper |
| 发布如何升级 | 快照、受管表、忽略表如何协作 |

## 核心原则

- 业务模块按插件组织，`plugin.json` 负责应用、菜单、view 和按钮权限清单，Provider 只负责命令、扫描路径、依赖和监听器等运行期装配。
- 标准 CRUD 复用四大基类，避免每个模块重复实现分页、软删、状态、数据范围和响应格式。
- 权限节点由 `plugin.json` 菜单/按钮清单和 `#[Auth]` 注解共同收敛，不手工维护散落的权限清单。
- 租户和数据范围默认 fail closed，拿不到上下文时不能返回全量数据。
- 发布升级使用快照和 dry-run，不依赖生产环境逐条跑开发迁移。

## 架构反模式

- 把业务模块都写进 `plugin/System`。
- 期望已发布二进制运行时远程安装或更新插件。
- Controller 直接写 SQL 或事务。
- Mapper 信任请求传入的字段名、排序、raw SQL。
- 前端隐藏按钮后后端不做 `#[Auth]` 校验。
- 只在列表做数据范围，详情、更新、删除、统计和选项接口不做；前端导出也必须复用受保护的列表接口。
- 在单例 Service 属性里保存用户、租户、Token 等请求态。
- 生产升级直接跑开发迁移，不做快照 dry-run。
