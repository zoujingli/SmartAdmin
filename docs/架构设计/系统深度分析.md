# 系统深度分析

本文基于当前 `plugin/System` 实现整理，重点解释系统装配、菜单同步、节点同步、菜单服务和看板数据为什么这样设计。

## 分析目标

本页不是 API 手册，而是解释 System 模块背后的设计选择：

- 为什么模块能力通过 `plugin.json` 与 Provider 分工装配。
- 为什么菜单和权限节点需要同步命令。
- 为什么菜单服务要承担路由、树结构和权限裁剪。
- 为什么看板数据不能简单返回全量统计。
- 为什么这些机制对开源项目长期维护有价值。

## 模块装配入口

`System` 模块通过 `plugin.json` 和 `Provider` 分工完成装配：

- `plugin.json`：声明应用、菜单、view、按钮权限、模块目录和模块目录卡片。
- Provider：注册注解扫描、依赖绑定、监听器和命令等运行期能力。
- 接口绑定：把登录、节点名称解析、日志写入绑定到具体 Service。

这意味着系统有哪些模块能力、菜单从哪里来、权限节点如何命名，并不是散落在业务代码里，而是由插件清单统一声明；运行期服务装配仍由 Provider 负责。

### Provider 的价值

| 价值 | 说明 |
|------|------|
| 模块自治 | `plugin.json` 声明菜单和页面，Provider 声明扫描、命令和监听器 |
| 降低耦合 | 新模块不需要修改 System 中央配置 |
| 易于发布 | 源码/CI 可统一校验插件清单、权限节点和运行期装配 |
| 易于文档化 | 文档可按 `plugin.json` 清单和 Provider 装配能力组织 |

新模块开发时，推荐通过自己的 `plugin.json` 声明菜单、view 和模块能力，通过 Provider 注册运行期能力，不要把业务模块配置硬编码进 System。

## 菜单基线同步链路

命令：`xadmin:menu:sync`

核心特点：

- `--dry-run`：只看差异，不落库。
- `--details`：输出新增和更新明细。
- `--json`：输出结构化报告，便于 CI 消费。
- `--exit-code`：按位掩码返回变化类型。

这套设计让菜单基线既能人工执行，也能纳入自动化发布检查。

### 菜单同步解决的问题

如果菜单完全靠后台页面手工维护，开源项目会遇到几个问题：

- fresh 初始化后菜单不完整。
- 不同部署环境菜单不一致。
- 前端路由变更后历史菜单漂移。
- 按钮权限 code 和后端接口 code 失配。
- 发布前难以知道菜单会发生什么变化。

`plugin.json` 加同步命令可以让菜单成为“可审查、可重复、可 dry-run”的唯一基线数据。

## 权限节点同步链路

命令：`xadmin:node:sync`

节点来源包括：

1. `#[Auth]` 注解扫描结果。
2. `plugin.json` 同步后的菜单 `code` 定义。

关键机制：

- 可通过 `--only=annotation/menu` 限制同步来源。
- 默认会先触发菜单同步（`plugin.json` 清单），可用 `--skip-menu-sync` 跳过。
- 同样支持 dry-run、details、json、exit-code。

项目把接口权限和菜单权限统一收敛到 `system_node`，避免接口注解和菜单按钮长期漂移。

### 节点同步解决的问题

权限节点来自两个世界：

- 后端接口注解：决定请求是否放行。
- 菜单和按钮 code：决定前端展示和角色授权。

如果两者不统一，常见表现是“按钮显示但接口 403”或“接口能访问但角色授权树看不到”。节点同步把两类来源统一到 `system_node`，角色只需要授权节点。

## 菜单服务关键行为

`MenuService` 不是简单 CRUD，它承担多个一致性责任：

- 入参兼容：把前端字段如 `parentId/path/permission` 统一映射到后端字段 `pid/route/code`。
- 树结构安全：禁止把菜单挂到自己或子孙节点下，防止拓扑环。
- 层级维护：变更父节点后自动刷新后代 `level`。
- 权限过滤：普通用户菜单和按钮权限按节点集合裁剪，超级管理员短路。
- 首页推导：根据可访问菜单树自动推导前端首页路径。

这使菜单体系同时满足后台管理树、前端路由树、权限节点过滤三类场景。

### 菜单服务边界

MenuService 负责菜单自身一致性，不负责业务接口授权的最终判断。最终接口授权仍由 `#[Auth]` 和 Auth 切面完成。

| 责任 | 是否属于 MenuService |
|------|----------------------|
| 菜单树维护 | 是 |
| 路由字段兼容 | 是 |
| 菜单状态和排序 | 是 |
| 用户菜单裁剪 | 是 |
| 接口请求是否放行 | 否 |
| 业务数据范围 | 否 |

## 系统看板数据链路

`DataService` 聚合系统看板与运行信息。它不只是拼接字段，而是按权限和数据范围输出：

- 统计项：用户、角色、菜单、部门、岗位、节点、日志、在线会话。
- 能力项：模块能力矩阵、公共能力、缓存驱动、权限策略描述。
- 运维项：工作台待办，例如异常日志、失败登录、租户状态、重复文件、在线会话健康。

关键点：

- 非超级管理员查看在线数据时，走用户数据范围过滤，避免统计接口泄露全量在线用户。
- 缓存清理分为全站清理和个人清理两个作用域。

### 看板数据为什么要后端聚合

如果前端直接请求多个模块接口拼装看板，会带来：

- 权限和数据范围在多个页面重复实现。
- 每个模块都暴露额外统计接口。
- 前端需要理解过多业务边界。
- 后续增加租户、缓存、告警时难以统一治理。

由 DataService 聚合可以统一权限、数据范围、缓存和展示结构，也便于运维页面增加风险提示。

## 日志和审计链路

System 模块的写操作普遍使用 `#[Logger]`，同时关键 Model 可以配置 `$logRules`。这形成三层审计：

| 层 | 内容 |
|----|------|
| 请求日志 | 记录请求路由、方法、响应码、客户端环境 |
| 操作日志 | 记录谁在什么接口做了什么操作 |
| 字段变更日志 | 记录关键字段从什么值变成什么值 |

关系变更如用户角色、角色节点、公告接收人，不一定触发模型字段事件，因此需要 Service 手动追加变更记录。

## 上传配置链路

文件模块把“上传记录”和“上传驱动配置”分开：

- 文件记录用于业务引用、下载、回收站、统计和去重。
- 上传配置用于控制 local、OSS、COS、Qiniu、Alist 等驱动。
- 上传流程通过 prepare、relay、chunk、part-sign、complete、abort 管理会话。

这种设计让业务模块不需要关心底层驱动，只保存文件 ID 或 URL。

## 租户和 System 的关系

System 模块提供用户、角色、菜单、文件、日志和租户管理等基础能力；租户隔离通过统一的租户上下文衔接：

- 登录后建立 TenantContext。
- 支持租户的 Model 通过 `tenant_id` 接入范围。
- 系统管理接口仍需要明确权限和数据范围。
- 平台侧租户管理不应和普通租户业务混用。

## 对开源项目的价值

- 可审计：同步命令有 dry-run、报告、退出码，方便接入 CI。
- 可扩展：模块目录与能力卡片通过注册表追加，不必改核心分发逻辑。
- 可治理：权限节点与菜单编码集中治理，降低长期维护成本。
- 可运维：看板数据包含风险提示和操作入口，不只是静态展示。

## 维护建议

- 新增模块时优先维护插件自己的 `plugin.json` 和 Provider，不要改 System 核心分发。
- 新增菜单或按钮时同步 `plugin.json`、菜单、节点、前端权限和接口文档。
- 新增统计项时确认数据范围和租户边界。
- 新增写操作时补 `#[Logger]` 和必要的 `$logRules`。
- 修改上传配置时同步脱敏规则和运维文档。
- 修改发布逻辑时同步 dry-run 输出、回滚说明和 release 检查。
