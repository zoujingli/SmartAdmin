# 文档维护规范

> 本页定义 SmartAdmin 公开文档的通用维护规则；接口字段、调试示例和 API 页面结构请同时遵守 [接口文档规范](./接口文档规范.md)。

最后更新：2026-05-08

## 维护原则

| 原则 | 说明 |
|---|---|
| 代码为准 | 文档必须来自当前源码、配置、迁移、前端页面和测试，不虚构未实现能力 |
| 同步更新 | 代码、菜单、权限、接口、部署或交互变更后，必须同步对应文档 |
| 单一入口 | 同一主题只保留一个主入口，补充专题必须指向主线文档，避免形成第二套口径 |
| 开源友好 | 文档面向陌生读者也要能读懂，必须写清前置条件、操作步骤、风险和源码来源 |
| 安全默认 | 示例不得暴露真实 Token、Secret、证书、手机号、身份证、生产域名和生产数据 |

## 文档分层

| 层级 | 目录 | 维护重点 |
|---|---|---|
| 入门体验 | `快速开始` | 环境、安装、启动、默认入口、常见问题 |
| 用户操作 | `用户教程` | 后台页面如何使用、关键按钮、风险确认和排障 |
| 系统能力 | `系统功能` | 核心能力边界、数据模型、权限、日志、租户、发布 |
| 扩展开发 | `开发指南` | 分层规范、插件开发、前端接入、编码约束 |
| 接口合同 | `接口参考` | 路由、权限、字段、示例、错误码、源码来源 |
| 架构说明 | `架构设计` | 整体设计、模块协作、关键机制和安全边界 |
| 部署发布 | `部署运维` | 开发/生产部署、Docs 静态站、发布升级、缓存日志 |
| 开源协作 | `开源协作` | 协议、贡献、行为准则、安全披露、路线图 |
| 维护说明 | `文档维护` | 文档规范、接口文档标准、检查和发布规则 |

## 必须更新文档的场景

| 变更类型 | 需要同步的文档 |
|---|---|
| 新增或变更 HTTP 接口 | `接口参考`、`完整接口清单`、相关用户教程或开发指南 |
| 新增或调整菜单/权限码 | `系统功能/认证权限与数据范围`、`接口参考`、插件 readme |
| 新增前端页面或交互变化 | `用户教程`、`系统功能`、插件 readme |
| 新增插件或资源根 | `开发指南/插件开发`、`接口参考`、插件 readme |
| 数据范围、租户、日志或安全策略变化 | `系统功能`、`架构设计`、`开发指南` |
| 部署、发布、构建、迁移策略变化 | `部署运维`、`运维手册`、`开源协作/发布流程` |
| 开源协议、贡献流程或支持渠道变化 | `开源协作`、根 `README`、插件 metadata |

## 内容质量标准

### 标题与结构

- 页面标题使用明确中文名，必要时保留英文术语，例如 `JSON-RPC 网关`。
- 标题层级不跳级，最多使用到四级标题；更深层级应拆表或拆页。
- 默认入口和目录页先给读者“我该看哪里”，再给背景说明。
- 大页面优先拆成目录页 + 字段级子文档，避免单页过长难以维护。

### 视觉和导航

- 文档站使用 `assets/devapi-docs.css`、`assets/devapi-scale.js`、`assets/devapi-module-nav.js`、`assets/devapi-sidebar-state.js` 和 `assets/devapi-mermaid.js` 统一 DevAPI 风格、DPI 缩放、顶部模块导航、左侧目录折叠记忆和流程图渲染。
- 默认入口、章节 README 和目录页优先采用“提示语 + 快速入口表 + 标准口径表 + 索引表”的结构，降低开源读者查找成本。
- 侧边栏一级分组使用 `* **分组名**`，二级入口使用中文业务名；顶部模块导航会读取一级分组，接口路径、源码类名和实现细节写在正文，不放到侧边栏标题里。
- 目录表建议控制在 4-5 列；接口字段表、错误码表和关键约束表按内容完整性优先。

### 表格与示例

- 目录表控制在 4-5 列，字段表按 `字段/类型/必填/说明` 或 `字段/类型/适用操作/必填/说明`。
- 所有 JSON 示例优先使用 `jsonc`，每个关键字段带中文注释。
- 响应示例保留标准业务响应壳：`code/info/data/path`。
- 高风险接口必须写关键约束、错误码和误用后果。
- 会员授权插件页面必须在 H1 前声明 `DOC_META`，并在首屏提示“会员授权 / 不随开源版源码分发”。
- 流程图统一使用 ` ```mermaid ` 代码块，优先采用 `flowchart LR` 展示横向主链路；文档站会通过 Mermaid 渲染图表。

### 链接与导航

- 新增页面必须同步章节 README 和 `docs/_sidebar.md`，一级分组调整会同步影响顶部模块导航。
- 普通文档内部链接可以使用相对路径；`_sidebar.md` 必须使用 docsify 根路径链接。
- 删除或移动页面时必须修复所有反向链接。
- 外部链接使用 HTTPS，并避免依赖不可公开访问的内部地址。
- `docs/index.html` 必须同时加载 docsify、搜索、复制、翻页、在线调试、Mermaid 和统一视觉资源；新增样式或脚本后同步 `xadmin:docs:check` 的必备资源校验。

## 目录整理规则

新增或调整文档目录时必须同步完成：

1. 判断内容属于主线文档、补充专题还是维护说明。
2. 主线文档优先更新默认入口、侧边栏和对应章节 README。
3. 补充专题必须在章节开头说明它补充哪个主线主题。
4. 无继续维护价值的重复正文应合并到主线文档后删除，不保留额外目录。
5. 调整目录后执行 `composer docs:check`，确保内部链接、接口案例和路由覆盖仍然有效。

## 接口文档维护流程

1. 从 Controller 注解提取真实路由、方法、权限码和操作日志。
2. 复核 Service、Mapper、Model、前端页面和插件清单，补齐字段、状态、租户和安全约束。
3. 更新接口页的“基本信息、请求字段、请求示例、响应字段、响应示例、错误码、关键约束、源码来源”。
4. 同步 `接口参考/完整接口清单.md` 的接口数量与路径。
5. 对每个接口补齐 `API_CASE`、`jsonc` 和 `api-test` 调试块。
6. 执行 `composer docs:check` 和 `git diff --check`。

## 提交前检查

```bash
composer docs:check
git diff --check
```

如本次修改包含 PHP、前端或构建逻辑，还需要按变更范围执行：

```bash
composer analyse
composer test
composer web:build
```

## 常见问题

| 问题 | 处理方式 |
|---|---|
| 文档和代码不一致 | 以代码为准修正文档；如代码行为不合理，先修代码再同步文档 |
| 接口示例过长 | 保留最小可调试字段，复杂字段放到字段表或补充说明 |
| 一个主题多处重复 | 保留主线文档，其他页面改为摘要和链接 |
| 开源协议口径冲突 | 以根 `LICENSE`、`NOTICE`、`docs/开源协作/授权协议.md` 和包 metadata 为准 |
