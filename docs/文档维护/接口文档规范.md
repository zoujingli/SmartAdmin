# 接口文档规范

> 面向开源用户、前后端开发者和测试联调人员的接口文档编写标准。目标是让每个接口页都能独立完成“理解用途、确认权限、复制示例、复现问题、回到源码”。

最后更新：2026-05-08

## 文档定位

接口参考是 SmartAdmin 对外公开的 HTTP 合同源头，内容必须来自当前代码实现，不从页面名称、方法名或历史习惯臆造接口。

| 文档对象 | 维护口径 |
|---|---|
| 后台 HTTP 接口 | 以 Controller 路由注解、`#[Auth]`、`#[Logger]`、Service 校验、Mapper 查询和 Model 字段为准 |
| 插件接口 | 以插件 `plugin.json`、Controller、Service、Mapper、Model 和前端表单共同校验为准 |
| 公开回调 | 以 Controller 路由、签名验签、消息解密、业务分发和响应格式为准 |
| 支付/资金接口 | 以 Service 事务、幂等规则、金额单位、通知验签和异常处理为准 |
| 文档示例 | 按源码真实行为推导，不为了文档示例修改业务逻辑或绕过安全校验 |

## 页面标准结构

每个包含真实 `方法与路径` 的接口页面，建议按下列结构维护。页面可以增加业务说明，但不能缺少核心联调信息。

| 顺序 | 标题 | 必填 | 内容要求 |
|---|---|---|---|
| 1 | 接口内容 | 是 | 说明接口组边界、适用模块、能力范围和不覆盖的场景 |
| 2 | 调试说明 | 是 | 说明 Base URL、认证头、Content-Type、分页、上传或回调调试方式 |
| 3 | 对接说明 | 是 | 说明前置配置、权限、调用顺序、幂等与安全边界 |
| 4 | 用途 | 是 | 用一段话描述当前接口解决什么业务问题 |
| 5 | 基本信息 | 是 | 方法与路径、Controller 方法、权限码、鉴权方式、操作日志 |
| 6 | 请求字段 | 是 | 分 `path/query/body/form/files/headers` 表格列出字段、类型、必填性和说明 |
| 7 | 请求示例 | 是 | 使用 `jsonc`，字段行必须带中文注释，去掉注释后可直接解析为 JSON |
| 8 | 响应字段 | 是 | 说明标准响应壳和 `data` 内部字段，标明必填/可选/条件返回 |
| 9 | 响应示例 | 是 | 保留项目标准响应壳 `code/info/data/path`，必要时给分页结构 |
| 10 | 错误码 | 是 | 说明 `401/403/404/500` 的业务边界和常见错误消息 |
| 11 | 关键约束 | 是 | 写清数据范围、租户隔离、敏感字段、幂等、金额单位、状态流转等 |
| 12 | 源码来源 | 是 | 列出 Controller、Service、Mapper、Model、前端页面或插件清单路径 |

## 接口目录页标准

较大的模块可以先做目录页，再拆分字段级子文档。目录页本身不替代接口页，必须清晰指向具体接口文档。

| 区块 | 建议内容 |
|---|---|
| 模块说明 | 当前模块解决的问题、后台菜单入口、插件名称、适用用户 |
| 接口内容 | 当前目录页覆盖的接口组数量、协议类型和数据边界 |
| 对接说明 | 认证方式、调试入口、前置配置、权限码检查方式 |
| 子文档 | 用 2 列表格列出子文档和覆盖范围 |
| 接口实例清单 | 用 3-5 列表格列出接口名称、方法与路径、字段级文档 |
| 阅读约束 | 说明目录页只做导航，字段和示例以具体接口页为准 |

## 字段表规范

字段表要让读者不看源码也能完成一次正确请求。

| 列名 | 要求 |
|---|---|
| 字段 | 使用代码字段名，嵌套字段使用点路径，例如 `body.extra.gateway_url` |
| 类型 | 使用 `string`、`integer`、`boolean`、`array`、`object`、`file`、`datetime` 等稳定类型 |
| 必填 | 写 `是`、`否`、`条件必填` 或 `返回时是`，不要留空 |
| 适用操作 | 多接口共用表时说明 `新增`、`更新`、`列表`、`回调` 等 |
| 中文说明 | 写业务语义、单位、取值范围、默认值、敏感性和边界条件 |

敏感字段必须明确说明：

- 写入字段是否加密保存。
- 列表或详情是否返回掩码。
- 操作日志是否排除或脱敏。
- 示例中不得出现真实密钥、证书、Token、手机号、身份证、生产域名或生产数据。

## JSONC 示例规范

所有可复制请求示例使用 `jsonc`。示例必须能被读者直接放入在线测试器或 HTTP 客户端改造使用。

```jsonc
{
  "method": "POST", // 必填：HTTP 方法，必须与基本信息一致
  "path": "/system/example/create", // 必填：请求路径，占位符需要替换为真实 ID
  "headers": { // 必填：请求头，权限接口必须携带 Bearer token
    "Content-Type": "application/json", // JSON 接口固定使用 application/json
    "Authorization": "Bearer <token>" // 后台登录后获取的访问令牌
  },
  "query": {}, // 必填：Query 参数，无参数也保留空对象
  "body": { // 必填：请求体，无请求体时使用空字符串或空对象
    "name": "示例名称" // 必填：业务名称，最多按 Service 校验规则填写
  }
}
```

维护要求：

1. 每个顶层字段必须有中文注释。
2. 嵌套对象的关键字段也要写中文注释。
3. 去掉注释后必须是合法 JSON。
4. 密码、Token、Secret、证书和生产数据使用 `<placeholder>`。
5. 上传接口使用 `content_type/form_data/files` 明确描述文件字段。
6. 回调接口必须说明原始 body、签名头、微信 query 参数和响应文本。

## 响应示例规范

SmartAdmin 标准业务响应使用 `code/info/data/path`。HTTP status 固定为 200，业务状态读取 `body.code`。

```jsonc
{
  "code": 200, // 业务响应码，成功固定为 200
  "info": "获取成功", // 业务提示文案，字段名固定为 info
  "data": { // 接口返回数据，列表接口通常包含 items/pageInfo
    "id": 1 // 示例 ID，不能使用真实生产数据
  },
  "path": "/system/example/info/1" // 当前请求路径
}
```

错误响应也要使用同一响应壳：

| code | 使用边界 |
|---|---|
| `401` | Token 缺失、过期、无效或刷新失败 |
| `403` | Token 有效但缺少操作权限 |
| `404` | 页面或 API 路由不存在 |
| `500` | 业务校验失败、状态不允许、数据不存在、外部接口失败等 |

## 源码复核清单

写接口文档前必须至少复核以下来源：

| 来源 | 复核重点 |
|---|---|
| Controller | prefix、Mapping、`#[Auth]`、`#[Logger]`、path 参数、公开入口 |
| Service | `_vali()` 规则、唯一性校验、事务、状态流转、外部 API 调用 |
| Mapper | 查询白名单、数据范围、租户范围、排序、列表后处理 |
| Model | `$fillable`、`$hidden`、类型转换、软删、关联、审计规则 |
| 前端页面 | 表单字段、默认值、二次确认、权限按钮、导出和上传行为 |
| 插件清单 | 菜单 route、view、权限 code、资源根、模块描述 |
| 测试用例 | 已覆盖的边界、异常和安全规则 |

## 开源可读性标准

开源文档既要能联调，也要能帮助新贡献者理解设计。新增或改写页面时遵守以下标准：

- 标题层级不跳级，目录页使用短标题，接口页使用动作化标题。
- 表格控制在 4-6 列；字段很多时拆成 `path/query/body/响应字段` 多张表。
- 先写业务语义，再写字段细节，避免一上来堆路由列表。
- 所有缩写第一次出现时给出中文解释，例如 APIv3、JSAPI、JSON-RPC。
- 高风险操作必须写“关键约束”和“误用后果”，例如退款、删除、发布、密钥轮换。
- 文档不得宣传未实现能力；规划中的能力放到路线图或 TODO，不写成可用接口。

## 提交流程

1. 先完成代码修改和就近测试。
2. 更新接口参考、用户教程或模块文档。
3. 同步 `_sidebar.md`、章节 README 和完整接口清单。
4. 如接口合同变化，补充更新日志。
5. 执行：

```bash
composer docs:check
git diff --check
```

6. 提交信息使用中文模块前缀，例如 `文档: 标准化微信接口说明`。
