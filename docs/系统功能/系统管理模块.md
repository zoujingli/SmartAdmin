# 系统管理模块

`plugin/System` 是后台系统核心模块，包含认证、组织权限、租户、文件、日志、通知、系统参数和系统数据。

## 模块边界

System 模块只承载通用后台能力，不承载具体业务域。开源使用者新增 CRM、订单、工单、财务等业务时，应优先新建插件模块，然后复用 System 提供的用户、角色、菜单、日志、文件和公告能力。

| 能力 | System 是否负责 | 说明 |
|------|-----------------|------|
| 登录、个人资料、密码 | 是 | 统一认证入口 |
| 用户、角色、菜单、部门、岗位 | 是 | 后台基础组织权限 |
| 租户管理 | 是 | 平台侧租户资料、状态、有效期和管理员开通 |
| 文件记录和上传配置 | 是 | 通用文件基础设施 |
| 操作日志和请求日志 | 是 | 通用审计基础设施 |
| 通知中心和收件箱 | 是 | 通用站内通知 |
| 系统参数和系统数据 | 是 | 通用配置、能力矩阵、缓存和工作台数据 |
| 业务订单、客户、工单 | 否 | 应放到独立业务插件 |

## 认证模块

负责登录、刷新 Token、获取用户信息、个人资料、界面偏好、密码修改、权限码和退出登录。

实现位置：

- `plugin/System/src/Controller/AuthController.php`
- `plugin/System/src/Service/UserService.php`
- `plugin/System/src/Service/AuthCacheService.php`

### 认证能力

| 能力 | 接口 | 说明 |
|------|------|------|
| 登录 | `POST /system/auth/login` | 校验账号密码并建立租户/用户上下文 |
| 刷新 Token | `POST /system/auth/refresh` | 访问令牌过期时换取新令牌 |
| 当前用户 | `POST /system/auth/profile` | 获取用户资料、角色、部门、岗位等信息 |
| 更新资料 | `PUT /system/auth/profile` | 修改昵称、头像、签名等个人资料 |
| 偏好设置 | `PUT /system/auth/preferences` | 保存界面配置 |
| 修改密码 | `PUT /system/auth/password` | 需要登录，日志排除密码字段 |
| 权限码 | `GET /system/auth/codes` | 前端按钮权限判断 |
| 退出登录 | `POST /system/auth/logout` | 清理登录态 |

认证模块不负责业务权限判断，业务接口仍需要自己的 `#[Auth]` 注解。

## 组织权限模块

| 模块 | 控制器 | 说明 |
|------|--------|------|
| 用户 | `UserController` | 用户 CRUD、关系分配、密码、统计 |
| 角色 | `RoleController` | 角色 CRUD、权限授权、角色用户 |
| 菜单 | `MenuController` | 菜单树、用户菜单、按钮权限 |
| 部门 | `DeptController` | 部门树、部门用户、排序 |
| 岗位 | `PostController` | 岗位 CRUD、状态、选项 |

这些模块都继承项目标准 CRUD 分层：Controller 只读参数和返回响应，Service 编排业务，Mapper 处理查询和数据范围，Model 定义表结构映射。

### 用户管理

用户是登录主体，也是操作日志、数据范围和租户上下文的重要来源。用户管理提供：

- 数据列表和回收站。
- 新增、编辑、软删除、恢复、彻底删除。
- 启用/禁用、排序、重置密码。
- 分配角色、部门、岗位。
- 统计和选项。

关键边界：

- 密码永远不返回、不明文记录日志。
- 用户列表、详情、更新、删除和选项都应受数据范围影响。
- 关系分配需要手动记录变更日志。

### 角色管理

角色聚合权限节点和数据范围。角色管理提供：

- 角色 CRUD、状态、回收站。
- 权限树读取和节点分配。
- 角色用户、角色选项和统计。

角色数据范围应谨慎授予。“全部数据”适合少数管理角色；普通岗位优先使用“本部门”“本部门及下级”或“仅本人”。

### 菜单管理

菜单同时服务前端路由和权限节点同步：

- 目录用于菜单分组。
- 菜单用于页面入口。
- 按钮用于页面内操作权限。

菜单字段中的 `route`、`component`、`redirect`、`keep_alive`、`hide_in_menu`、`hide_in_tab` 会影响前端路由表现。按钮 `code` 必须与后端 `#[Auth]` code 一致。

### 部门和岗位

部门用于组织树和数据范围，岗位用于用户标签和筛选。部门删除前需要检查子部门和用户占用，岗位删除前需要检查用户关联，避免产生悬挂关系。

## 系统数据

`DataController` 提供工作台统计、系统信息、系统配置、能力矩阵、待办事项、界面元信息和业务缓存清理；`SettingController` 提供系统参数读取和保存。

系统配置通过 `system_data` 存储，适合存放可维护的运行配置，不适合存放大体积业务正文。

### 系统数据接口

| 能力 | 接口 | 用途 |
|------|------|------|
| 工作台统计 | `/system/data/index` | 首页统计和概览 |
| 系统信息 | `/system/data/info` | 运行环境、版本、基础信息 |
| 系统配置 | `/system/data/config` | 读取和保存可维护配置 |
| 能力矩阵 | `/system/data/capabilities` | 展示系统模块能力 |
| 待办事项 | `/system/data/todos` | 工作台待办 |
| 界面元信息 | `/system/data/ui-meta` | 登录后读取低敏 UI 元信息 |
| 清理缓存 | `/system/data/clear-cache` | 清理业务缓存 |
| 系统参数 | `/system/setting/info` | 读取和保存系统参数 |

登录页和应用启动优先使用公开的 `/system/auth/ui-meta`。`/system/data/ui-meta` 要求有效 Token，但可接受 System 后台或 Project 应用等实现 `UserModelInterface` 的登录主体；它只返回低敏展示元信息，不开放系统业务数据。

系统配置不适合存储对象存储明文密钥、JWT 密钥、数据库密码等高敏感配置；这类配置应由 `.env` 或安全配置系统维护。

## 文件、日志和公告

| 模块 | 控制器 | 说明 |
|------|--------|------|
| 文件 | `FileController` | 文件记录、下载、回收站、统计、上传配置、上传流程、去重 |
| 日志 | `LogsActionController`、`LogsChangeController` | 操作日志列表、变更日志、详情、清理、统计、指标、分析 |
| 通知 | `NoticeController` | 公告 CRUD、发布、状态、收件箱、未读、已读、归档 |

这些模块是后台系统的运行支撑能力：

- 文件模块负责存储抽象和上传记录，不应承载具体业务附件规则。
- 日志模块负责审计和排查，不应记录明文敏感数据。
- 公告模块负责站内通知，不应替代复杂消息队列或外部通知平台。

## 前端对应

System 页面由 `plugin/System/plugin.json` 声明，真实 view 由 `plugin.view_root` 指向 `plugin/System/stc/view`；Web 只提供通用运行库、布局、公共组件和共享依赖。主要页面目录：

- `plugin/System/stc/view/user`
- `plugin/System/stc/view/role`
- `plugin/System/stc/view/menu`
- `plugin/System/stc/view/dept`
- `plugin/System/stc/view/post`
- `plugin/System/stc/view/tenant`
- `plugin/System/stc/view/file`
- `plugin/System/stc/view/logs/action`
- `plugin/System/stc/view/logs/change`
- `plugin/System/stc/view/notice`
- `plugin/System/stc/view/setting`
- `plugin/System/stc/view/data`

当前 System API 封装仍由 Web 公共运行库提供；新增业务插件的 API service、私有组件和页面建议放在各自显式配置的 `plugin.view_root` 下，并通过相对路径自管理。

## 二次开发建议

- 新增通用后台资源时，优先参考用户、角色、部门、岗位的 CRUD 实现。
- 新增关系分配时，参考用户角色/部门/岗位和角色节点分配的日志处理。
- 新增文件业务时，复用文件上传接口并在业务表中保存文件 ID 或 URL。
- 新增审计字段时，配置 `$logRules`，为中文名、枚举值和单位提供可读说明。
- 新增工作台统计时，优先通过 `DataService` 聚合，不要让前端直接请求多个内部接口拼装核心数据。

## 排查清单

| 问题 | 检查 |
|------|------|
| 用户看不到菜单 | 用户角色、角色节点、菜单状态、菜单同步 |
| 用户能看到按钮但接口 403 | 按钮 code 和 `#[Auth]` code 是否一致 |
| 列表数据不对 | Mapper 数据范围、租户上下文、搜索条件 |
| 日志缺失 | 写操作是否加 `#[Logger]`，模型是否配置 `$logRules` |
| 文件上传失败 | 上传配置、场景、驱动、大小、MIME、分片会话 |
| 公告未收到 | 公告是否发布，接收人关系是否生成，是否已归档 |
