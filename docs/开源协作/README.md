# 开源协作

本章面向外部使用者、贡献者和维护者，说明如何反馈问题、提交改动、参与版本发布和报告安全问题。

## 协作目标

SmartAdmin 的开源协作围绕三个目标：

- 保持核心后台能力稳定：权限、租户、数据范围、日志、上传、发布。
- 降低二次开发成本：提供清晰分层、接口参考、前端接入和模块模板。
- 保持文档和实现同步：让用户能依靠文档完成安装、使用、开发和部署。

## 授权与公开边界

| 类型 | 说明 | 贡献与分发口径 |
|------|------|----------------|
| 开源能力 | 随公开 SmartAdmin 仓、公开 Composer 包或公开插件包提供 | 可按 Apache-2.0 和对应开源协议使用、修改、提交 PR |
| 会员授权插件 | 文档站可公开展示接口与使用说明，但源码、ZIP 和安装使用需会员授权或内部交付 | Issue 和文档建议可公开提交，源码改动在授权交付链路处理 |
| 私有交付插件 | 客户专有能力或内部商业模块 | 不进入公开仓和公开 Release；如需同步文档，应避免泄露客户数据、密钥和专有流程 |

文档中出现 Website、Smart、Project、Asset、Points、WechatService 等“会员授权”能力时，只代表能力说明和接口合同可见，不代表公开开源版已经包含对应源码或可安装 ZIP。

## Issue 建议

提交 issue 时建议包含：

| 类型 | 建议内容 |
|------|----------|
| 缺陷 | 版本、环境、复现步骤、期望结果、实际结果、日志或截图 |
| 文档问题 | 页面链接、错误内容、建议修改方向 |
| 功能建议 | 使用场景、目标用户、是否适合通用后台能力 |
| 部署问题 | 操作系统、PHP/Swoole、MySQL、Redis、Node、命令输出 |
| 安全问题 | 不要公开细节，按安全披露流程处理 |

## 对贡献者的期望

- 问题描述可复现，尽量提供环境、版本、日志、截图或最小复现步骤。
- PR 描述包含背景、方案、影响范围和验证命令。
- 涉及接口、权限、部署、发布、上传、日志和前端交互的变更必须同步文档。
- 不提交 `runtime`、`vendor`、`build`、`public`、`web/dist`、前端构建压缩包、本地 `.env` 和 IDE 文件。

## PR 合并关注点

维护者会优先看：

- 改动是否属于项目通用能力。
- 是否遵守 Controller、Service、Mapper、Model 分层。
- 是否覆盖权限、数据范围、租户隔离和日志审计。
- 是否同步前端 API、页面、按钮权限和路由。
- 是否更新 docs。
- 是否提供匹配的验证命令。
- 是否引入不必要依赖或破坏升级路径。

## 开源文档维护

当前文档站默认进入 [快速开始](../快速开始/README.md)，主章节包括快速开始、用户教程、系统功能、开发指南、接口参考、架构设计、部署运维、开源协作。项目采用 Apache-2.0 协议发布，文档站是独立静态站，维护规则见 [Docs 静态站](../部署运维/Docs静态站.md)。接口文档必须精确到请求/响应字段、中文注释、JSONC 示例和可调试配置，不能用路径列表替代字段合同。

## 维护者职责

- 保持主分支可构建、可测试、可启动。
- 对权限、发布、上传、租户、安全相关改动保持更高审查标准。
- 引导贡献者补齐测试和文档。
- 及时更新路线图和发布说明。
- 对安全问题优先响应，并控制公开细节节奏。
