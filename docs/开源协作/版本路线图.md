# 版本路线图

> 路线图用于表达维护方向，不构成严格发布时间承诺。

## 路线图原则

- 优先保证核心链路可靠：认证、权限、租户、日志、上传、发布。
- 优先补齐通用后台能力，不追逐单一业务场景。
- 优先改进可验证性：测试、dry-run、静态检查、文档检查。
- 优先保持后向兼容，破坏性变更必须有迁移说明。

## 短期计划（1~2 个版本）

- 持续保持 `System` 模块文档与代码实现一一对应（菜单、权限、日志、上传、公告）
- 完善 `xadmin:menu:sync` / `xadmin:node:sync` 的 CI 接入示例
- 增强发布检查结果可读性（dry-run 输出分组与风险提示）
- 补齐接口参考中的请求体、响应结构和前端关联
- 增强 docs 静态检查，覆盖更多内部链接和章节完整性
- 梳理上传配置脱敏、分片会话和对象存储错误提示

## 中期计划（3~5 个版本）

- 完善租户场景下的数据范围与在线会话可视化
- 增加系统配置、上传配置、权限策略的回归测试覆盖
- 优化前后端接口一致性检查流程，减少手工对齐成本
- 沉淀新模块生成或模板化接入方案
- 提供更清晰的角色授权差异预览
- 完善发布升级报告，增强 SQL 风险分级
- 增强日志归档、分析和异常告警能力

## 长期方向

- 保持 `Library -> System -> Tenant` 分层稳定，降低跨模块耦合
- 沉淀模块接入模板，减少新业务模块接入成本
- 持续优化文档可读性与开源协作流程
- 建立更稳定的扩展点，让业务模块通过 `plugin.json`、Provider 和 Registry 分工接入系统能力
- 提升私有化交付体验，包括构建、发布、回滚、配置检查和诊断报告
- 逐步完善多版本文档和升级指南

## 暂不优先

以下内容短期不作为优先方向：

- 与通用后台能力无关的垂直业务模块。
- 大规模替换现有前端框架或后端分层。
- 引入无法被私有化部署稳定使用的重型外部依赖。
- 绕过发布快照机制的生产升级方案。

## 如何参与路线图

如果你希望推动某项能力，请在 issue 或 PR 中说明：

- 使用场景和目标用户。
- 为什么它适合通用后台项目。
- 是否影响权限、租户、日志、上传、发布或前端体验。
- 计划如何验证。
- 需要同步哪些文档。
