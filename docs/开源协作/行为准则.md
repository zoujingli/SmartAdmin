# 行为准则

## 目标

建设一个专业、友好、可持续协作的开源社区。

社区讨论应围绕项目目标展开：构建可靠、可扩展、可私有化部署的通用后台系统。技术争议可以直接，但必须基于事实、代码、复现步骤和可执行建议。

## 我们倡导

- 尊重不同背景与经验的参与者
- 基于事实与代码讨论问题，避免人身化表达
- 对新贡献者保持耐心，提供可执行反馈
- 反馈问题时尽量提供环境、版本、日志、截图和复现步骤
- Review 时指出具体文件、接口、风险和改进建议
- 对安全问题使用私密披露渠道，而不是公开可利用细节

## 良好协作示例

- “这个接口缺少 `#[Auth]`，普通用户可能越权访问，建议补权限码并同步接口文档。”
- “这个 PR 影响上传配置，请补充脱敏测试和 Docs 静态站说明。”
- “我在 PHP 8.4 + Swoole 6.2 下复现了启动失败，错误日志如下。”
- “这个功能更像业务模块，建议作为独立插件而不是放进 System。”

## 不可接受行为

- 侮辱、骚扰、歧视、威胁等不当言行
- 恶意刷屏、引战、故意破坏讨论秩序
- 在公开渠道披露敏感信息或可利用漏洞细节
- 未经授权发布他人账号、密钥、Token、日志、数据库、截图中的敏感信息
- 明知有破坏性仍鼓励他人执行未备份的删除、清空、强制升级操作
- 反复提交与项目目标无关、无法维护的大范围改动

## 处理流程

1. 维护者收到举报后先进行事实核查
2. 视严重程度给出提醒、限制或移除协作权限
3. 重大问题保留记录，并在必要范围内公告处理结果

## 处理措施

视情况可能采取：

| 措施 | 场景 |
|------|------|
| 友好提醒 | 初次偏离讨论主题或信息不完整 |
| 要求修改 | PR 描述、代码、文档或测试不符合要求 |
| 暂停讨论 | 讨论失焦、重复争执或情绪化升级 |
| 限制权限 | 持续破坏协作秩序或泄露敏感信息 |
| 安全隔离 | 涉及漏洞利用、密钥泄露或数据风险 |

## 举报建议

请提供时间、场景、相关链接和必要截图，便于快速判断与处理。

如果举报内容包含密钥、Token、漏洞 PoC、用户数据或生产日志，请先脱敏，并优先通过安全披露渠道联系维护者。
