# 发布流程

## 流程目标

确保版本发布过程可追踪、可回滚、可复盘，降低线上风险。

发布流程同时服务两类场景：

- 开源版本发布：面向社区用户发布 tag、更新文档和说明。
- 私有化交付发布：面向具体环境执行构建、升级、验证和回滚准备。


## GitHub TAG 自动同步

SmartAdmin 只从私有 `SmartAdmin` 仓库手工打 TAG。推送 `v*` TAG 后，`.github/workflows/release.yml` 会执行三段自动化：

1. 从 `plugin/Library` 导出 `SmartAdminLibrary`，检查通过且目标仓有 diff 时提交到 `master` 并同步同名 TAG。
2. 按开源白名单导出 `SmartAdmin`，排除 `plugin/Library`、私有/商用插件、运行产物和敏感文件，检查通过且有 diff 时提交到 `master` 并同步同名 TAG。
3. 扫描 Developer 仓内所有带 `plugin.json` 的可安装插件，生成 `<plugin-code>-<version>-<tag>.zip` 与 `plugins-manifest.json`。开源插件 ZIP 可附加到 `SmartAdmin` Release，私有/商用插件 ZIP 只附加到 `SmartAdmin` 私有 Release 或内部存储。

目标仓无 diff 时不提交、不推送、不移动 TAG。私有/商用插件不发布 Composer 远程包；ZIP 内 `composer.json` 仅用于安装后的本地 path autoload。

## 标准流程

1. **需求冻结**：确认本次版本范围，不再接入高风险新需求。
2. **质量校验**：执行静态检查、测试、前端构建、发布检查。
3. **文档同步**：更新功能说明、接口参考、运维说明、更新日志。
4. **预发布验证**：在预发布环境完成回归和升级演练。
5. **正式发布**：按发布窗口执行升级并观察核心指标。
6. **发布复盘**：记录问题与改进项，纳入下个迭代。

## 发布前清单

| 类别 | 检查项 |
|------|--------|
| 范围 | 本次包含哪些功能、修复、文档、配置和数据库变化 |
| 权限 | 是否新增或修改 `#[Auth]` code、菜单、按钮、角色授权 |
| 数据库 | 是否涉及结构变化、受管表、忽略表、发布快照 |
| 前端 | 是否涉及路由、API service、表单、按钮权限 |
| 插件 | 是否涉及本地插件 `composer.json`、Provider、`plugin.json`、插件前端或授权信息 |
| 运维 | 是否涉及缓存、上传、日志、对象存储、环境变量 |
| 安全 | 是否涉及密钥、Token、敏感字段、越权风险 |
| 文档 | 是否同步用户教程、接口参考、部署运维、FAQ |

## 建议命令

```bash
composer analyse
composer test
composer web:build
composer build
composer release:check
composer docs:check
```

`composer build` 只复用并校验已有 `web/dist`；如果本次涉及插件页面、`routes.ts`、`auth-entry.ts` 或主前端变化，必须先执行 `composer web:build` 或由 CI 提供等价前端产物。

涉及数据库发布时额外保留：

```bash
composer release:backup
composer release:upgrade:dry-run
sh bin/start-swoole xadmin:release:upgrade --dry-run --json
```

## 发布门禁建议

- 未完成文档同步不允许进入发布窗口
- 未完成升级 dry-run 不允许执行正式升级
- 涉及权限、迁移、缓存策略变更必须有回滚说明
- 涉及破坏性 SQL 不允许默认使用 `--force`
- 涉及安全修复必须准备升级说明和临时缓解建议
- 涉及上传驱动或密钥配置必须确认脱敏和备份策略

## 版本号建议

可按语义化版本理解：

| 版本 | 说明 |
|------|------|
| Patch | 修复缺陷、文档更新、小范围兼容优化 |
| Minor | 新增向后兼容能力、模块、页面或接口 |
| Major | 破坏性接口、配置、数据库或部署方式变化 |

如果项目尚未稳定到 1.0，仍建议在发布说明中明确兼容性和升级影响。

## 正式发布步骤

1. 拉取发布代码或部署发布包。
2. 确认 `.env`、runtime、上传目录和发布快照在正确位置。
3. 执行数据库升级 dry-run。
4. 备份数据库和上传文件。
5. 执行默认升级。
6. 如已评审破坏性 SQL，再人工执行 `--force`。
7. 重启 Swoole 进程或切换到新发布实例。
8. 验证登录、菜单、用户、角色、文件、日志、公告、租户和关键业务。
9. 观察错误日志、请求失败率、数据库和 Redis 连接。

## 回滚流程

| 场景 | 操作 |
|------|------|
| 仅代码问题 | 切回上一版本代码或镜像，重启服务 |
| 受管数据异常 | 使用 `xadmin:release:restore --backup=<id>` 恢复 |
| 前端或插件资源异常 | 回滚上一发布包，或重新执行 `composer web:build` 后用 `composer build` 生成新发布包 |
| 配置异常 | 恢复 `.env` 或配置备份，重启服务 |
| 上传异常 | 回滚上传配置，必要时恢复对象存储配置 |

回滚后仍需要重新验证核心路径，并记录复盘。

## 发布后复盘

建议记录：

- 实际发布时间和影响时长。
- 是否执行了 `--force`。
- 备份 ID 和回滚验证结果。
- 发布期间出现的问题。
- 文档和自动化检查需要改进的地方。
