# 发布流程

## 流程目标

确保版本发布过程可追踪、可回滚、可复盘，降低线上风险。

发布流程同时服务两类场景：

- 开源版本发布：面向社区用户发布 tag、更新文档和说明。
- 私有化交付发布：面向具体环境执行构建、升级、验证和回滚准备。


## GitHub TAG 自动同步

SmartAdmin 生态只从私有 `SmartAdminDeveloper` 仓库手工打 TAG。推送 `v*` TAG 后，`.github/workflows/release.yml` 会执行四段自动化：

1. 从 `plugin/Library` 导出 `SmartAdminLibrary`，检查通过且目标仓有 diff 时提交到 `master` 并同步同名 TAG。
2. 从 `plugin/Builder` 导出 `SmartAdminBuilder`，检查通过且目标仓有 diff 时提交到 `master` 并同步同名 TAG。
3. 按开源白名单导出 `SmartAdmin`，排除 `plugin/Library`、私有/商用插件、运行产物和敏感文件，检查通过且有 diff 时提交到 `master` 并同步同名 TAG。
4. 扫描 Developer 仓内所有带 `plugin.json` 的可安装插件，生成插件 ZIP 与 `plugins-manifest.json`。插件 ZIP 统一命名为 `plugin-<plugin-code>-<plugin-version>.zip`（如 `plugin-system-1.0.0.zip`），不混入发布 TAG，便于同一插件版本在不同发布批次中保持稳定文件名。开源插件 ZIP 可附加到 `SmartAdmin` Release，私有/商用插件 ZIP 只附加到 `SmartAdminDeveloper` 私有 Release 或内部存储。

目标仓无 diff 时不提交、不推送、不移动 TAG。私有/商用插件不发布 Composer 远程包；ZIP 内 `composer.json` 仅用于安装后的本地 path autoload。

## 仓库职责与文档口径

公开文档中需要同时出现四个仓库，但含义不同：`SmartAdminDeveloper` 是来源说明，不是普通用户入口；`SmartAdmin` 才是公开主项目和有效使用入口；`SmartAdminLibrary` 与 `SmartAdminBuilder` 是独立 Composer 基础包。

| 仓库 | 可见性 | 面向对象 | 分发方式 | 文档描述重点 |
|------|--------|----------|----------|--------------|
| `zoujingli/SmartAdminDeveloper` | Private | 生态维护者、内部开发团队 | 不发布公开 Composer 包；私有 Release 保存全量插件 ZIP | 唯一全量开发源、唯一手工 TAG 发布源，维护开源核心、基础包源码、构建器源码、公开插件、私有/商用插件和发布 Actions |
| `zoujingli/SmartAdmin` | Public | 普通用户、社区贡献者、二次开发者 | `composer require zoujingli/smartadmin` 或克隆公开仓 | 开源主项目，包含可运行后台框架、Web 通用壳、公开插件、文档和测试，不包含私有/商用插件源码 |
| `zoujingli/SmartAdminLibrary` | Public | SmartAdmin、插件和二次开发项目 | `composer require zoujingli/smart-admin-library` | 开源基础库，说明 Core CRUD、权限、数据范围、租户、日志、发布升级和源码期插件 ZIP 管理能力 |
| `zoujingli/SmartAdminBuilder` | Public | 发布构建流程、私有化交付流水线 | `composer require zoujingli/smart-admin-builder` | 开源构建器，说明 Phar/SFX 打包、配置 AST 改写、源码加固、前端资源归档和二进制产物生成能力 |

开发边界固定为：只在私有 `SmartAdminDeveloper` 仓库维护完整生态，包含开源核心、基础包、构建器、公开插件、私有/商用插件和发布流水线；正常用户只需要使用公开 `SmartAdmin` 仓库或 `zoujingli/smartadmin` 包即可运行和二次开发。若用户启用插件安装机制，可在 SmartAdmin 源码环境通过 ZIP 安装或更新插件，更新同一插件使用 `xadmin:plugin:install <zip> --force`，然后执行 Composer 更新和前端构建；生产二进制仍只运行已构建进包内的插件。

## TAG Release 信息自动生成

`SmartAdminDeveloper` 的私有 Release 信息由 `.github/workflows/release.yml` 在发布流水线内生成；`SmartAdmin`、`SmartAdminLibrary`、`SmartAdminBuilder` 会随导出携带 `.github/workflows/release-notes.yml`，目标仓收到同名 TAG 后可独立生成或更新 GitHub Release。

自动发布信息包含：

- 仓库独立定位、开源属性、授权协议、Composer 包名和发布范围。
- 针对仓库代码能力的进度描述：按文件变更归类到基础库、构建器、Web 壳、系统插件、微信插件、发布自动化、测试和文档等能力域。
- 基于上一个 `v*` TAG 到当前 TAG 的提交记录，兼容 `feat:`、`fix:` 以及 `发布:`、`文档:`、`前端:` 等中文前缀分组。
- GitHub 仓库 Stars、Forks、Watchers、Open Issues、Release 资产累计下载和当前 TAG 资产下载统计。
- Packagist Composer 包下载统计：总下载、月下载、日下载和 Favorites。

统计口径说明：GitHub 不提供源码 ZIP/clone 总下载量，Release 信息只统计 GitHub Release 资产下载量和 Packagist Composer 下载量；私有仓或未发布 Composer 的仓库会显示不适用或暂不可用。

## 标准流程

1. **需求冻结**：确认本次版本范围，不再接入高风险新需求。
2. **质量校验**：执行静态检查、测试、前端构建、发布检查。
3. **文档同步**：更新功能说明、接口参考、运维说明、更新日志。
4. **预发布验证**：在预发布环境完成回归和升级演练。
5. **正式发布**：按发布窗口执行升级并观察核心指标。
6. **发布复盘**：记录问题与改进项，纳入下个迭代。

## 发布前清单

| 类别 | 检查项 |
|------|--------|
| 范围 | 本次包含哪些功能、修复、文档、配置和数据库变化 |
| 权限 | 是否新增或修改 `#[Auth]` code、菜单、按钮、角色授权 |
| 数据库 | 是否涉及结构变化、受管表、忽略表、发布快照 |
| 前端 | 是否涉及路由、API service、表单、按钮权限 |
| 插件 | 是否涉及本地插件 `composer.json`、Provider、`plugin.json`、插件前端或授权信息 |
| 运维 | 是否涉及缓存、上传、日志、对象存储、环境变量 |
| 安全 | 是否涉及密钥、Token、敏感字段、越权风险 |
| 文档 | 是否同步用户教程、接口参考、部署运维、FAQ |

## 建议命令

```bash
composer analyse
composer test
composer web:build
composer build
composer release:check
composer docs:check
```

`composer build` 只复用并校验已有 `web/dist`；如果本次涉及插件页面、`routes.ts`、`auth-entry.ts` 或主前端变化，必须先执行 `composer web:build` 或由 CI 提供等价前端产物。

涉及数据库发布时额外保留：

```bash
composer release:backup
composer release:upgrade:dry-run
sh bin/start-swoole xadmin:release:upgrade --dry-run --json
```

## 发布门禁建议

- 未完成文档同步不允许进入发布窗口
- 未完成升级 dry-run 不允许执行正式升级
- 涉及权限、迁移、缓存策略变更必须有回滚说明
- 涉及破坏性 SQL 不允许默认使用 `--force`
- 涉及安全修复必须准备升级说明和临时缓解建议
- 涉及上传驱动或密钥配置必须确认脱敏和备份策略

## 版本号建议

可按语义化版本理解：

| 版本 | 说明 |
|------|------|
| Patch | 修复缺陷、文档更新、小范围兼容优化 |
| Minor | 新增向后兼容能力、模块、页面或接口 |
| Major | 破坏性接口、配置、数据库或部署方式变化 |

如果项目尚未稳定到 1.0，仍建议在发布说明中明确兼容性和升级影响。

## 正式发布步骤

1. 拉取发布代码或部署发布包。
2. 确认 `.env`、runtime、上传目录和发布快照在正确位置。
3. 执行数据库升级 dry-run。
4. 备份数据库和上传文件。
5. 执行默认升级。
6. 如已评审破坏性 SQL，再人工执行 `--force`。
7. 重启 Swoole 进程或切换到新发布实例。
8. 验证登录、菜单、用户、角色、文件、日志、公告、租户和关键业务。
9. 观察错误日志、请求失败率、数据库和 Redis 连接。

## 回滚流程

| 场景 | 操作 |
|------|------|
| 仅代码问题 | 切回上一版本代码或镜像，重启服务 |
| 受管数据异常 | 使用 `xadmin:release:restore --backup=<id>` 恢复 |
| 前端或插件资源异常 | 回滚上一发布包，或重新执行 `composer web:build` 后用 `composer build` 生成新发布包 |
| 配置异常 | 恢复 `.env` 或配置备份，重启服务 |
| 上传异常 | 回滚上传配置，必要时恢复对象存储配置 |

回滚后仍需要重新验证核心路径，并记录复盘。

## 发布后复盘

建议记录：

- 实际发布时间和影响时长。
- 是否执行了 `--force`。
- 备份 ID 和回滚验证结果。
- 发布期间出现的问题。
- 文档和自动化检查需要改进的地方。
