# 常见问题

本页按“启动、登录、权限、上传、日志、发布、文档”整理常见问题。排查时建议先确认当前操作的是开发环境、预发环境还是生产环境，避免把本地缓存或测试账号问题误判为系统缺陷。

## 项目采用什么开源协议？

SmartAdmin 采用 Apache License 2.0 发布。你可以在遵守 Apache-2.0 条款的前提下使用、修改和再分发项目源码；再分发时需要保留许可证、版权声明、修改说明和必要的 NOTICE 信息。

前端部分基于 Vue Vben Admin 等第三方组件改造，第三方组件继续遵守其原始许可证。项目根目录的 `NOTICE` 统一记录主要第三方归属说明。

## 后端启动失败

先检查 `.env` 中数据库、Redis 和端口配置。确认依赖后执行：

```bash
composer setup
sh bin/start-swoole start
```

如果本机没有 Swoole 扩展，确认当前平台是否有项目内置运行时；没有内置运行时时，需要本机 PHP 安装 Swoole。

### 如何定位启动错误？

1. 看终端第一屏错误，通常能直接看到缺扩展、配置缺失、端口占用。
2. 看 `runtime/logger`，确认是否有数据库、Redis、依赖注入或注解扫描错误。
3. 执行 `./bin/swoole-macos-a64 -m` 查看内置运行时扩展。
4. 确认 `.env` 是否从 `.env.example` 复制并填写。
5. 如果改过 Provider、注解、配置文件，重启 Swoole。

### 端口被占用怎么办？

检查常用端口：

```bash
lsof -i :9501
lsof -i :5666
lsof -i :18100
```

后端端口通过 `.env` 调整；前端端口通过前端 dev server 配置调整；docs 端口在 `composer docs:serve` 脚本中固定为 `18100`。

## 前端无法登录

排查顺序：

1. 后端 `http://127.0.0.1:9501` 是否可访问。
2. 前端 API 地址是否指向正确后端。
3. `/system/auth/login` 是否返回 Token。
4. `/system/auth/profile` 和 `/system/auth/codes` 是否正常。

账号密码错误是业务失败，按项目规范返回 `500`；Token 缺失或失效才返回 `401`。

### 登录成功但马上退出

可能原因：

- 前端保存 Token 失败。
- `/system/auth/profile` 返回异常。
- `/system/auth/codes` 或用户菜单接口返回 401/403。
- 系统时间差导致 Token 过期判断异常。
- 刷新 Token 接口失败。

先打开浏览器 Network 面板，看登录后的第一个失败接口。

## 菜单或按钮不显示

执行同步命令后重新登录：

```bash
sh bin/start-swoole xadmin:menu:sync --details
sh bin/start-swoole xadmin:node:sync --details
```

仍不显示时检查菜单 code、`#[Auth]` code、角色授权和前端按钮权限是否一致。

### 菜单点击空白

常见原因：

- 菜单 `component` 不存在或路径不匹配。
- 前端路由未注册或动态路由解析失败。
- 目录菜单缺少 `redirect`。
- 页面加载接口 403，导致页面初始化失败。

先确认菜单 route 和 component，再看浏览器控制台和 Network。

## 角色权限修改后不生效

权限缓存可能仍是旧版本。更新角色节点后应重新登录；如果是系统级权限变更，执行缓存清理或触发权限版本更新。

### 为什么前端隐藏按钮还要后端校验？

前端隐藏按钮只是体验控制。用户可以绕过前端直接请求 API，所以后端接口必须依赖 `#[Auth]` 做最终校验。任何影响数据读取、修改、导出、清理的接口都不能只靠前端控制。

## 数据范围不符合预期

排查顺序：

1. 用户是否有多个角色。
2. 每个角色的 `scope` 是什么。
3. 用户是否分配了部门。
4. 目标接口是否使用标准 Mapper 数据范围。
5. 详情、更新、删除、导出、统计是否也接入数据范围。
6. 是否存在超级管理员或全部数据角色。

如果是新模块，重点检查 Mapper 的归属字段，不能让请求参数决定 `deptField`、`userField` 或 raw SQL。

## 上传失败

先看 [文件上传与存储](../系统功能/文件上传与存储.md)。重点检查：

- 上传通道是否启用。
- 文件大小、后缀、MIME 是否在允许范围内。
- 分片大小和驱动能力是否匹配。
- 远端存储密钥是否配置完整。
- 上传会话是否由同一登录用户创建和完成。

### 上传 413 或文件过大

同时检查：

- Nginx `client_max_body_size`。
- PHP 上传限制。
- 前端上传组件限制。
- 系统上传 runtime 配置。
- 对象存储单文件或分片限制。

### 对象存储 403

通常是密钥、Bucket、Region、Endpoint、Domain 或临时签名错误。先用最小文件验证上传，再测试下载 URL 和 CDN 回源。

## 日志里看不到明文密钥

这是预期行为。密码、Token、Secret、Key、Cookie、Authorization、上传签名等敏感字段不得明文入库或进入日志。

### 操作日志缺失

检查：

- 写操作是否加 `#[Logger]`。
- AOP/切面是否生效。
- 是否通过标准响应异常链路返回。
- 关系变更是否需要 Service 手动追加变更记录。
- 日志表是否被清理或归档。

## 发布升级前必须做什么

至少执行：

```bash
composer release:check
composer docs:check
```

涉及数据库发布时，还要保留 `xadmin:release:upgrade --dry-run` 输出，并确认 `backup_tables`、`ignore_tables` 配置符合预期。

### 什么时候使用 `--force`？

只有 dry-run 已明确显示破坏性 SQL，并且你已经确认备份、维护窗口、影响表和回滚方案后才使用。不要把 `--force` 写进默认发布脚本。

### 为什么生产升级不直接跑 migrations？

迁移适合开发期建表和 fresh 初始化。生产发布需要知道目标库和发布快照之间的差异、受管数据表、忽略表、备份路径和破坏性 SQL，所以项目采用 DBAL 快照机制。

## 文档页面点击无反应

先强刷浏览器，因为 docsify 会缓存 `_sidebar.md`、Markdown 和 `assets/` 静态资源。如果仍异常，运行：

```bash
composer docs:check
```

顶部模块导航由 `_sidebar.md` 一级分组自动生成；侧边栏必须使用 `/章节/页面`。

### docs 服务如何启动？

```bash
composer docs:serve
```

访问：

```text
http://127.0.0.1:18100
```

这是本地预览服务，不适合公网生产。生产请参考 [Docs 静态站](../部署运维/Docs静态站.md)。

### 为什么不把 docs 接到后端 `/docs`？

docs 是独立静态站，便于开源托管、版本化、单独部署和 CDN 缓存。后端只负责业务 API 和管理端，不承担文档站运行时。

## 如何提一个容易合并的 PR？

1. 范围小，聚焦一个问题。
2. 保持 Controller、Service、Mapper、Model 分层。
3. 权限、日志、租户、数据范围同步考虑。
4. 后端、前端、文档一起对齐。
5. 写清验证命令和结果。
6. 不混入无关格式化和本地环境文件。

更多要求见 [贡献指南](./贡献指南.md)。
