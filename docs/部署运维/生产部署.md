# 生产部署

生产部署建议把后端 API、前端静态资源、docs 静态站、release 安装包和运行备份拆开管理。后端由 Swoole 常驻进程或已发布 SFX 二进制提供服务，前端由构建产物或后端兜底静态能力提供，docs 独立静态站部署，不挂到 Hyperf 后端 `/docs` 路由。

私有化交付优先使用 `composer build` 生成的 `build/system*` 二进制；源码部署仅适合自管服务器或需要在目标机保留源码、Composer 和 Node.js 的场景。

## 常规源码部署

1. 准备 PHP 8.4、Swoole 6、MySQL、Redis、Node.js 和 pnpm。
2. 首次获取源码可执行 `composer create-project --no-dev zoujingli/smartadmin SmartAdmin`；如果使用 GitHub 克隆源码，则进入目录后执行 `composer install --no-dev --optimize-autoloader`。
3. 配置 `.env`，生产环境建议使用 Redis 缓存。
4. 执行 `composer web:build` 构建前端，或由 CI 放置等价 `web/dist` 产物。
5. 执行 `composer setup` 初始化或同步菜单节点。
6. 使用进程管理器启动 `./bin/smart.php start`。

源码部署统一使用 `bin/smart.php` 作为命令入口。部署流水线、systemd、Supervisor 和文档示例都应直接指向 `bin/smart.php`；启动服务时必须显式携带 `start`，避免无参数入口进入开发 watch。

### 推荐目录

```text
/srv/smartadmin/
  app/                 # 当前发布版本源码
  shared/.env          # 生产环境配置
  shared/runtime/      # 日志、缓存、上传、本地快照
  releases/            # 历史版本，可用于回滚
  docs/                # docsify 静态站，可独立挂载
```

如果采用简单部署，也可以直接在项目目录运行，但至少要把 `.env`、`runtime`、上传目录和数据库备份放入可靠备份策略。

### 环境配置

生产 `.env` 应至少确认以下项目：

| 配置 | 说明 |
|------|------|
| `APP_ENV` | 使用生产环境标识，建议 `prod` |
| `APP_NAME` | 应用进程标识和站点名称 |
| `APP_WORKER_PORT` / `APP_WORKER_NUMS` | Swoole 监听端口和 Worker 数；监听地址当前固定为 `0.0.0.0` |
| `APP_DEBUG_XSQL` / `APP_DEBUG_XLOG` | 生产建议关闭 SQL/请求调试输出 |
| `APP_TITLE` / `APP_API_URL` | 前端运行时标题和 API 前缀；Nginx 直接托管 `web/dist` 时建议 `APP_API_URL=/api` |
| `DB_*` | 数据库连接、账号和库名 |
| `REDIS_*` / `CACHE_DRIVER` / `CACHE_PREFIX` | 缓存、Token 状态、权限缓存和多实例隔离配置 |
| 登录令牌密钥 | 当前由 `config/autoload/jwt.php` 提供默认开发配置；生产应按安全要求替换为独立密钥方案 |
| 上传驱动配置 | local/oss/cos/qiniu/alist 的访问域名和密钥 |

敏感配置不要写入文档、日志、仓库或前端构建产物。

## 前端静态资源

前端构建产物在 `web/dist`，它是本地/CI 生成产物，不提交 Git。源码部署时需要先执行 `composer web:build` 或由 CI 放置等价产物；二进制部署时 Phar 内只携带 `storage/extra/web-dist.zip`，首次 `start` 或手动执行 `xadmin:website:publish` 后发布到 `public/`。`SiteMiddleware` 可以在未匹配 API 路由时返回前端资源，并动态生成 `_app.config.js` 注入 API 地址和标题。

部署方式有两种：

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| 后端托管 | 由 Hyperf 未匹配 API 路由时返回前端资源 | 单体部署、内网部署、简单私有化 |
| Nginx 托管 | Nginx 直接指向 `web/dist`，API 反代到 Swoole | 高并发静态资源、CDN、前后端分离部署 |

无论采用哪种方式，都要确认 `_app.config.js` 中的 API 地址、站点标题和资源路径符合生产域名。

插件前端页面、`routes.ts` 和 `auth-entry.ts` 都会在 `composer web:build` 阶段编入 `web/dist`。SmartAdminLibrary 提供的 `xadmin:plugin:*` 只服务源码/CI 插件 ZIP 和备份流程，其中 backup 默认只备份代码，`--with-data` 才包含插件自有表；生产环境不支持在已发布二进制内动态安装、更新或移除插件，新增或升级插件后应重新构建前端产物和发布包。

## 反向代理建议

生产环境建议 Nginx 终止 TLS，再转发到 Swoole 端口：

```nginx
location / {
    proxy_pass http://127.0.0.1:9501;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

如果前端由 Nginx 直接托管，推荐在后端 `.env` 中设置 `APP_API_URL=/api`，让 `_app.config.js` 注入统一 API 前缀；Nginx 再把 `/api/` 反代到 Swoole 并剥离该前缀。这样可以避免 `/project/**` 这类插件前台页面路由与同名前台 API 前缀互相抢占：

```nginx
upstream smartadmin_backend {
    server 127.0.0.1:9501;
}

server {
    listen 443 ssl http2;
    server_name admin.example.com;

    root /srv/smartadmin/app/web/dist;
    index index.html;

    # _app.config.js 由后端按 .env 动态生成，不能使用 web/dist 内的静态文件替代。
    location = /_app.config.js {
        proxy_pass http://smartadmin_backend;
        proxy_set_header Host $host;
        proxy_set_header Authorization $http_authorization;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # APP_API_URL=/api 时，前端请求 /api/system/...，此处转发为后端真实 /system/...。
    location /api/ {
        proxy_pass http://smartadmin_backend/;
        proxy_set_header Host $host;
        proxy_set_header Authorization $http_authorization;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

如果生产仍保持 `APP_API_URL=/`，就不能只代理 `/api/`，需要按实际接口前缀精确代理 `/system`、`/website`、`/material`、`/wechat-client`、`/wechat-service` 以及各插件 API；注意不要把插件 SPA 页面路由整段转发给后端，否则刷新 `/project/login`、`/project/task` 等页面时会变成后端 404。Website 和 Material 开放接口业务请求使用 `Authorization: Bearer <access_token>`，反代时应保留 `Authorization`，跨域直连时 CORS 只需额外允许 `Authorization` 与 `Content-Type` 等标准头。

### 开放接口反代要求


- 保留 `Authorization`，否则业务接口无法解析 Bearer Token。
- 保留 `X-Real-IP` 和 `X-Forwarded-For`，否则 IP 白名单、分钟限流和 `last_used_ip` 审计会失真。
- 不要把 AppKey 下发到普通浏览器页面；浏览器前台应通过 BFF、SSR 或同域服务端代理换取 Token。

当 `APP_API_URL=/` 且需要直接暴露开放接口前缀时，可增加：

```nginx
location ^~ /website/ {
    proxy_pass http://smartadmin_backend;
    proxy_set_header Host $host;
    proxy_set_header Authorization $http_authorization;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

location ^~ /material/ {
    proxy_pass http://smartadmin_backend;
    proxy_set_header Host $host;
    proxy_set_header Authorization $http_authorization;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

docs 静态站建议单独域名或路径部署，示例见 [Docs 静态站](/部署运维/Docs静态站)。

## 进程管理

可以使用 systemd、supervisor 或容器编排启动后端进程。关键要求是：

- Swoole 进程异常退出后自动拉起。
- 发布时先健康检查新版本，再切换流量。
- 日志输出进入统一采集系统或文件轮转。
- 不在生产环境直接使用 `composer docs:serve` 或 PHP 内置服务器承载公网流量。
- 不在生产环境使用无参数 `./bin/smart.php` 或 `./bin/smart.php watch`；无参数入口默认进入 watch，只服务源码期本地调试。

如果部署介质丢失可执行权限，先在发布目录执行：

```bash
chmod +x ./bin/smart.php
```

systemd 示例（当前 `start` 为前台进程，交由 systemd 管理生命周期）：

```ini
[Unit]
Description=SmartAdmin
After=network.target

[Service]
Type=simple
WorkingDirectory=/srv/smartadmin/app
ExecStart=/srv/smartadmin/app/bin/smart.php start
Restart=always
RestartSec=3
User=www-data
Group=www-data
KillSignal=SIGTERM
TimeoutStopSec=30

[Install]
WantedBy=multi-user.target
```

当前应用命令不提供 `reload` / `stop` 子命令；发布切换、配置变更或代码变更时使用 `systemctl restart`、Supervisor restart 或容器滚动替换。

## 发布前检查

```bash
composer install --no-dev --optimize-autoloader
composer web:build
composer docs:check
composer release:check
```

如果本次发布包含数据库结构或必要数据变化，应先在预发环境执行 `xadmin:release:backup --with-data`，再执行 `xadmin:release:restore --install --dry-run`，确认 `backup_tables`、`ignore_tables`、破坏性 SQL 和备份路径符合预期。

## 回滚建议

1. 保留最近数个 `releases` 目录或镜像标签。
2. 发布前备份 `.env`、数据库、上传文件，并生成 `runtime/backup` 运行备份。
3. 代码回滚先切换到旧版本并重启 Swoole。
4. 如果数据库升级已执行，需要使用最新全量运行备份执行 `xadmin:release:restore --with-data`。
5. 回滚后抽查登录、菜单、用户列表、文件上传和关键业务接口。

## 注意事项

- `.env` 不提交仓库。
- 生产环境不建议使用 File 缓存。
- 发布前执行 `composer release:check`。
- 日志、上传文件、`runtime/backup` 运行备份应纳入备份策略。
- 对象存储密钥、登录令牌签名密钥、数据库密码不得进入前端配置和 docs。
- 删除、清空、彻底删除、发布升级等高风险操作应保留审计日志。
